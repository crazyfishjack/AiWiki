# 投标助手 AI 上下文工程设施设计调研

## 调研问题

投标助手 AI 上下文之外的工程设施设计，包括企业资料库、响应矩阵、规则引擎、版本管理等核心组件的架构与最佳实践。

## 摘要

本文档基于四工具调研（Tavily + Exa + Firecrawl + Jina）收集了 20 个候选信源，精读 10 个，成功获取 4 个完整内容。通过二次精读补充了关键来源，最终编译为中文调研报告。

- **发现候选信源**：20 个
- **精读信源数量**：10 个
- **精读成功数量**：4 个（初次）+ 3 个（二次精读）
- **精读失败数量**：6 个（Firecrawl 402 欠费）

## 核心结论

### 1. 上下文工程是 AI 系统的架构基础 [可信度：高]

上下文工程（Context Engineering）已从提示工程演进为现代 AI 系统的核心架构基础。它关注的是构建一个**动态系统**，在正确的时间、以正确的格式，为 LLM 提供完成任务所需的所有正确信息和工具。[来源: https://www.53ai.com/news/tishicijiqiao/2025072179506.html]

**关键转变**：
- 从"制作一个字符串"到"构建一个系统"
- 从静态模板到动态组装
- 从单轮交互到多轮有状态系统
- 从关注模型能力到关注信息流水线质量

### 2. 上下文窗口是高度结构化的数据对象 [可信度：高]

上下文窗口不是单一文本块，而是由多种组件构成的结构化数据：

| 上下文类别 | 组件 | 描述 |
|----------|------|------|
| **指令性** | 系统提示、用户输入、少样本示例 | 指导 LLM 行为 |
| **状态性** | 聊天历史、长期记忆、暂存区 | 维持跨会话连贯性 |
| **外部知识** | 检索数据（RAG） | 连接外部知识源 |
| **工具驱动** | 工具定义、工具响应 | 与外部系统交互 |
| **结构化** | 输出模式、输入数据 | 约束格式提高可靠性 |

[来源: https://docs.uipath.com/zh-CN/agents/automation-cloud/latest/user-guide/best-practices-for-context-engineering]

### 3. 版本感知 RAG 是投标文档管理的关键 [可信度：高]

VersionRAG 研究表明，标准 RAG 在处理版本化文档时准确率仅 58%，而版本感知方法可达 90%。投标文档频繁更新（补遗、澄清、版本迭代），必须显式建模版本关系和变更追踪。[来源: https://arxiv.org/html/2510.08109]

**核心能力**：
- 版本序列和关系建模
- 显式和隐式变更检测
- 版本特定内容边界
- 基于意图的路由检索

### 4. 企业级 AI 需要完整的工程设施栈 [可信度：高]

仅靠提示词和 LLM 无法构建可靠的投标助手，需要完整的工程设施：
- **企业资料库**：资质、人员、业绩、产品的结构化存储
- **响应矩阵**：招标要求到投标章节的映射
- **规则引擎**：废标项、评分公式、资质匹配校验
- **版本管理**：文档版本、模板版本、知识库版本
- **溯源机制**：每个结论必须可追溯到原文位置

[来源: https://www.53ai.com/news/tishicijiqiao/2025072179506.html]

### 5. 上下文压缩与记忆架构是长任务的关键 [可信度：中]

投标任务通常跨越数百轮交互，需要：
- **短期记忆**：当前对话历史
- **长期记忆**：跨会话持久化（用户偏好、项目摘要）
- **暂存区**：多步骤任务的中间结果
- **上下文压缩**：提取、过滤、摘要技术减少 token 消耗

[来源: https://www.53ai.com/news/tishicijiqiao/2025072179506.html]

### 6. 评估体系必须独立于生成模型 [可信度：高]

传统 NLP 指标（BLEU、ROUGE）无法评估事实准确性和上下文相关性。需要专门的评估框架（如 RAGAs）：
- **上下文精确率**：检索内容的相关性
- **上下文召回率**：是否找到所有必要信息
- **忠实度**：答案是否基于上下文而非幻觉
- **答案相关性**：答案是否切题

[来源: https://www.53ai.com/news/tishicijiqiao/2025072179506.html]

## 内容整理

### 一、上下文工程的架构组件

#### 1.1 上下文窗口解剖

上下文窗口应被视为 API 请求负载，而非简单字符串拼接。

**指令性上下文**：
- **系统提示**：设定角色、目标、行为约束
- **用户输入**：任务触发点
- **少样本示例**：引导格式和推理模式

**时序/状态上下文**：
- **短期记忆**：当前对话历史（滚动窗口或摘要缓冲区）
- **长期记忆**：跨会话持久化（向量数据库实现）
- **暂存区**：多步骤任务的临时工作空间

**外部知识上下文**：
- **RAG 检索**：从知识库动态获取事实依据

**工具驱动上下文**：
- **工具定义**：描述可用工具、参数、预期输出
- **工具响应**：执行结果反馈到上下文

**结构化上下文**：
- **输出模式**：JSON Schema 约束响应格式
- **结构化输入**：JSON 等紧凑格式提高效率

#### 1.2 上下文管理技术

**上下文压缩**：
- **提取**：LLM 提取相关片段（精度高但延迟高）
- **过滤**：二元决策保留/丢弃文档（更快但粒度粗）
- **摘要**：生成简洁摘要（大幅减少 token 但可能丢失细节）
- **高级技术**：Gisting、IC-Former 等新兴压缩模型

**记忆架构**：
- **短期记忆**：对话缓冲区（滚动窗口或摘要）
- **长期记忆**：向量数据库存储关键信息
- **暂存区**：避免用冗长思考链污染主上下文

**上下文隔离**：
- **沙箱化**：工具调用在独立环境执行，只返回摘要
- **状态隔离**：结构化对象（Pydantic 模型）而非扁平文本

### 二、企业资料库设计

#### 2.1 资料库核心模块

**资质库**：
- 资质证书（名称、编号、等级、有效期、扫描件）
- 认证证书（ISO、行业认证）
-  licenses（软件许可、专利）

**人员库**：
- 基本信息（姓名、身份证、联系方式）
- 资质证书（职称、执业资格、培训证书）
- 业绩关联（参与过的项目）
- 有效期管理（证书到期提醒）

**业绩库**：
- 项目基本信息（名称、金额、时间、地点）
- 合同扫描件
- 验收证明
- 业主评价
- 关联人员

**产品库**：
- 产品信息（名称、规格、参数）
- 检测报告
- 认证证书
- 价格信息

#### 2.2 资料库工程设施

**结构化存储**：
- 关系型数据库（PostgreSQL/MySQL）存储元数据
- 对象存储（MinIO/S3）存储扫描件
- 向量数据库（Milvus/Pinecone）存储语义嵌入

**版本管理**：
- 资料变更历史
- 版本对比
- 回滚机制
- 审批流程

**有效期管理**：
- 到期提醒（提前 30/60/90 天）
- 自动标记过期资料
- 替换流程

**权限控制**：
- 基于角色的访问控制（RBAC）
- 资料分级（公开、内部、机密）
- 审计日志

### 三、响应矩阵设计

#### 3.1 响应矩阵结构

响应矩阵是投标助手的核心产物，映射招标要求到投标响应：

| 字段 | 说明 |
|------|------|
| requirement_id | 要求唯一标识 |
| requirement_text | 原文内容 |
| source_file | 来源文件 |
| source_page | 页码 |
| source_section | 章节 |
| response_chapter | 响应章节 |
| required_materials | 所需材料 |
| company_match | 企业匹配结果 |
| match_status | 匹配状态（完全匹配/部分匹配/不匹配） |
| responsible_person | 责任人 |
| risk_level | 风险等级（高/中/低） |
| deadline | 截止时间 |
| status | 状态（待处理/进行中/已完成） |

#### 3.2 响应矩阵生成流程

1. **招标解析**：提取资格要求、评分标准、废标项、采购需求
2. **要求分类**：按类型分组（资格、评分、废标、技术、商务）
3. **材料映射**：每类要求对应所需材料清单
4. **企业匹配**：从企业资料库检索匹配资料
5. **缺口识别**：标记缺失或过期的资料
6. **风险评估**：基于匹配度和重要性评估风险
7. **责任分配**：分配给相应负责人
8. **进度跟踪**：跟踪完成状态

#### 3.3 响应矩阵工程设施

**规则引擎**：
- 资格匹配规则（资质等级、有效期、业绩金额）
- 评分公式计算（价格分、技术分、商务分）
- 废标项检查（否定条件、组合条件）
- 暗标检查（身份泄露检测）

**溯源机制**：
- 每个字段必须带证据（evidence_quote）
- 来源定位（文件、页码、章节、block_id）
- 置信度标注（high/medium/low）
- 人工复核标记（needs_human_review）

**冲突检测**：
- 跨章节要求冲突
- 资质有效期冲突
- 业绩重复使用检测

### 四、规则引擎设计

#### 4.1 规则引擎架构

**规则层**：
- **废标规则**：否定条件、组合条件、替代条件
- **评分规则**：价格公式、技术评分、商务评分
- **资质规则**：等级匹配、有效期校验、专业匹配
- **暗标规则**：身份泄露检测（页眉页脚、水印、元数据）

**校验层**：
- **JSON Schema 校验**：输出格式合规性
- **业务规则校验**：日期、金额、分值合理性
- **一致性校验**：跨章节、跨文件一致性
- **完整性校验**：必填字段、必附材料

**重试层**：
- **解析失败重试**：JSON 格式错误、Schema 不合规
- **校验失败重试**：业务规则不满足、一致性检查失败
- **人工升级**：多次重试失败后升级给人工

#### 4.2 规则引擎实现

**规则定义**：
```python
# 废标规则示例

    {
        "id": "DQ-001",
        "type": "negative",
        "condition": "not_have_qualification",
        "params": {
            "qualification_type": "建筑工程施工总承包",
            "min_level": "二级"
        },
        "consequence": "disqualified",
        "evidence_required": True
    },
    {
        "id": "DQ-002",
        "type": "combination",
        "conditions": [
            {"type": "not_have", "target": "安全生产许可证"},
            {"type": "expired", "target": "安全生产许可证"}
        ],
        "logic": "or",
        "consequence": "disqualified"
    }
]
```

### 五、版本管理设计

#### 5.1 版本管理范围

**文档版本**：
- 招标文件版本（初稿、补遗、澄清）
- 投标文件版本（草稿、修改稿、终稿）
- 模板版本（章节模板、格式模板）

**知识库版本**：
- 企业资料库版本（资质更新、人员变更）
- 规则库版本（评分规则、废标规则）
- 提示词版本（抽取提示、生成提示）

**配置版本**：
- 模型配置（模型选择、参数设置）
- 流水线配置（解析流程、校验流程）

#### 5.2 版本管理工程设施

**版本存储**：
- Git 存储文档和配置版本
- 数据库存储元数据和变更历史
- 对象存储存储快照

**版本对比**：
- 文本差异对比（diff）
- 结构化数据对比（JSON diff）
- 语义差异对比（LLM 辅助）

**版本回溯**：
- 任意版本恢复
- 变更历史查询
- 变更原因记录

**版本发布**：
- 审批流程
- 版本号管理（语义化版本）
- 发布说明

## 适用场景

### 适合采用的项目类型

1. **大型工程投标**：招标文件数百页，涉及多专业、多章节
2. **频繁投标的企业**：需要复用企业资料，提高效率
3. **合规要求高的行业**：需要严格的溯源和审计
4. **多团队协作**：需要责任分配和进度跟踪
5. **历史项目多的企业**：需要积累和利用历史经验

### 典型用户画像

- **投标经理**：统筹整个投标过程，需要全局视图
- **技术人员**：编写技术方案，需要快速查找相关要求
- **商务人员**：准备商务文件，需要资质和业绩匹配
- **审核人员**：审查投标文件，需要一致性检查
- **管理层**：决策投标策略，需要风险评估

---

## 不适用场景

### 可能过度工程的情况

1. **简单小额采购**：招标文件简单，人工即可处理
2. **一次性投标**：没有复用价值，投入产出比低
3. **高度定制化项目**：每次都是全新内容，难以模板化
4. **紧急投标**：时间紧迫，来不及建立完整体系
5. **资料匮乏的新企业**：没有足够的历史资料积累

### 投入产出比低的场景

- 年投标次数少于 10 次
- 招标文件平均少于 50 页
- 团队规模小于 5 人
- 没有专职的投标管理人员

## 风险与约束

### 技术风险

1. **LLM 幻觉**：模型可能编造不存在的资质、业绩或公式
   - **缓解**：强制证据链、规则校验、人工复核

2. **OCR 错误**：扫描件识别错误导致信息遗漏
   - **缓解**：多表示保真、人工抽检、置信度标注

3. **检索质量**：RAG 检索不到相关信息或检索到错误信息
   - **缓解**：混合检索、重排序、评估体系

4. **版本混乱**：文档版本管理不当导致使用过期信息
   - **缓解**：版本管理、有效期提醒、变更追踪

### 工程风险

1. **数据质量**：企业资料库数据不完整或过期
   - **缓解**：数据治理、定期更新、质量检查

2. **系统集成**：与现有系统（OA、ERP）集成复杂
   - **缓解**：标准化接口、渐进式集成

3. **用户接受度**：团队成员不愿使用新系统
   - **缓解**：培训、渐进式推广、收集反馈

4. **维护成本**：系统需要持续维护和更新
   - **缓解**：自动化运维、文档完善、团队建设

### 合规风险

1. **数据安全**：投标信息泄露
   - **缓解**：权限控制、加密存储、审计日志

2. **暗标泄露**：无意中暴露身份信息
   - **缓解**：暗标检查、元数据清理、水印检测

3. **知识产权**：使用他人模板或内容
   - **缓解**：原创性检查、版权管理

## 对于可信度较高的来源逐来源总结

### 来源 1: VersionRAG 论文

**核心内容**：
- 标准 RAG 在处理版本化文档时准确率仅 58%
- VersionRAG 通过显式建模版本关系达到 90% 准确率
- 识别三种查询类型：内容检索、版本查询、变更检索
- 使用层级图结构捕获版本序列、内容边界和变更
- 索引时 token 消耗比 GraphRAG 少 97%

**可用事实**：
- 版本化文档 QA 需要专门的检索策略
- 版本混淆（Version Conflation）是核心问题
- 隐式变更检测是难点（标准 RAG 0%，VersionRAG 60%）
- 图结构比纯向量检索更适合版本管理

**局限**：
- 仅在技术文档上测试，未验证投标文档场景
- 隐式变更检测准确率仍较低（60%）
- 需要人工标注版本关系

**适合入库的知识点**：
- 版本感知 RAG 的架构设计
- 三种查询类型的路由策略
- 版本图结构的构建方法

### 来源 2: UiPath 上下文工程最佳实践

**核心内容**：
- 上下文锚定（Context Grounding）是连接智能体与企业知识的关键
- 索引设置最佳实践：版本化命名、提取模式选择、定期同步
- 结果数量和阈值校准：平衡检索率和 token 消耗
- 常见错误排除：未找到结果、阈值过高、上下文溢出、信息过时

**可用事实**：
- 使用版本化索引名称（如 `HR-Policies-2025-Q3`）
- 基本模式适合纯文本，高级模式适合扫描件
- 默认阈值为 0，提高阈值可减少噪声但可能遗漏相关信息
- 评估运行应包括真实世界的提示和回归测试

**局限**：
- 主要针对通用企业场景，未专门针对投标
- 未详细讨论规则引擎和响应矩阵

**适合入库的知识点**：
- 上下文锚定的工程实践
- 索引管理和版本控制
- 评估和调试方法

### 来源 3: 53AI 上下文工程教程

**核心内容**：
- 上下文工程是管理 LLM"工作记忆"的艺术与科学
- 上下文窗口由 5 类组件构成：指令性、状态性、外部知识、工具驱动、结构化
- RAG 是基石技术，高级模式包括混合搜索、重排序、多跳 RAG、智能体 RAG
- 上下文压缩技术：提取、过滤、摘要、高级模型
- 记忆架构：短期记忆、长期记忆、暂存区
- 评估框架：RAGAs 提供细粒度指标

**可用事实**：
- 提示工程关注"指令"，上下文工程关注"信息"
- 上下文组件的顺序影响性能
- 混合搜索结合关键词搜索和语义搜索
- 智能体 RAG 是 RAG 的最前沿形态
- "大海捞针"问题：LLM 难以回忆深埋中间的信息

**局限**：
- 主要是理论介绍，缺乏投标场景的具体实践
- 未详细讨论企业资料库和响应矩阵

**适合入库的知识点**：
- 上下文工程的完整理论框架
- RAG 的高级模式和陷阱
- 上下文管理技术

### 证据 2: VersionRAG 的性能对比

> VersionRAG achieves 90% accuracy, outperforming naive RAG (58%) and GraphRAG (64%). VersionRAG reaches 60% accuracy on implicit change detection where baselines fail (0-10%).

[来源: https://arxiv.org/html/2510.08109]
