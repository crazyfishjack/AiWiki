# 投标助手设计方案

## 一、设计目标与定位

### 1.1 核心定位

投标助手不是"通用大模型写作文"，而是面向招投标场景的**垂直生产系统**。其核心价值在于：

- **先读标，再写标**：从招标文件解析出发，而非从零开始写作
- **有据可查**：每个结论必须能追溯到招标文件原文或企业资料
- **人机协同**：AI 负责执行层（读标、生成、检查、复用），人负责决策层（投标策略、真实性确认、商务决策、最终责任）
- **闭环交付**：输出可提交、可审计、可复盘的正式投标文件

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **响应矩阵驱动** | 以"招标要求→投标响应"的映射关系为核心，贯穿全流程 |
| **分阶段生成** | 先解析、再规划、后生成，禁止一键整本生成 |
| **证据链优先** | 每个字段、每句正文都必须有来源证据 |
| **规则兜底** | 日期、金额、分值、资质有效期等硬校验由规则引擎完成 |
| **人工终裁** | 高风险项必须进入人工复核队列，AI 不替代最终责任 |

### 2.2 核心数据模型

#### 2.2.1 招标文件解析对象（10 类）

| 对象 | 说明 | 关键字段 |
|------|------|----------|
| `ProjectInfo` | 项目基础信息 | 名称、编号、标段、采购人、代理、预算、采购方式 |
| `Timeline` | 时间节点 | 报名、答疑、保证金、投标截止、开标、交付 |
| `EligibilityRequirement` | 资格要求 | 资质、业绩、人员、财务、信誉、联合体 |
| `EvaluationCriteria` | 评分标准 | 技术分、商务分、价格分、公式、证明材料 |
| `MandatoryClause` | 废标/否决条款 | 触发条件、后果、风险等级 |
| `ProcurementRequirement` | 采购需求 | 范围、数量、参数、功能、服务、验收 |
| `BidDocumentRequirement` | 投标文件格式 | 目录、签章、装订、页码、封装 |
| `BlindBidRequirement` | 暗标规则 | 禁止信息、禁止载体、格式统一要求 |
| `MaterialRequirement` | 材料清单 | 营业执照、资质证书、人员证书、业绩证明 |
| `ContractTerm` | 合同条款 | 付款、质保、违约、交付、保密 |

#### 2.2.2 响应矩阵（ResponseMatrix）

响应矩阵是投标助手的**核心中间产物**，字段设计：

```json
{
  "requirement_id": "REQ-SCORE-003",
  "requirement_type": "evaluation_criteria",
  "requirement_title": "项目实施方案评分",
  "original_text": "实施方案完整、可行得 10 分...",
  "source": {
    "file": "招标文件.pdf",
    "page": 31,
    "section": "第三章 评标办法",
    "block_id": "p31-t02-r05"
  },
  "is_mandatory": true,
  "risk_level": "high",
  "response_target": {
    "bid_section": "技术方案 > 项目实施方案",
    "materials": ["历史实施方案", "项目进度模板"]
  },
  "company_match": "matched",
  "match_status": "完全匹配",
  "responsible_person": "技术负责人",
  "status": "已确认",
  "confidence": "high",
  "needs_human_review": true
}
```

## 三、关键流程设计

### 3.1 标准工作流（11 步）

```
0. 项目建档与权限设定
   └─ 创建项目工作区，设定密级、权限、审计规则

1. 文件包归集与版本管理
   └─ 收集招标公告、招标文件、补遗、澄清、附件、图纸、平台说明
   └─ 建立版本表，处理补遗/澄清与原文件的冲突覆盖

2. 文档解析与版面还原
   └─ 原生解析优先（Word/Excel/可复制 PDF）
   └─ OCR 兜底（扫描件、图片、盖章页、复杂表格）
   └─ 输出：Markdown + HTML Table + Layout JSON + Page/BBox

3. 章节切分与语义分块
   └─ 按章节、表格、语义切分为独立 chunk
   └─ 每个 chunk 记录：ID、章节路径、页码、来源块、类型

4. 结构化抽取（10 类对象）
   └─ 分任务抽取：项目/时间/资格/评分/废标/需求/暗标/材料/合同/格式
   └─ 每条字段必须带：evidence_quote、source、confidence、needs_human_review

5. 响应矩阵生成
   └─ 将抽取结果映射到投标章节、企业资料、责任人、风险状态
   └─ 输出：响应矩阵 + 缺失材料清单 + 风险清单

6. 企业资料匹配（RAG）
   └─ 从资质库、人员库、业绩库、产品库匹配证明材料
   └─ 硬条件过滤：资质等级、有效期、主体一致性
   └─ 缺料时输出"缺失清单"，不得编造

7. 目录规划与分章节生成
   └─ 按招标文件格式和评分点生成目录
   └─ 每章独立生成，携带：章节目标、招标要求、授权资料、禁止事项
   └─ 生成内容必须引用 evidence_id，缺料处标注"【需补资料】"

8. 专业附件生成
   └─ 技术偏离表、商务偏离表、资质证明清单、人员/业绩表
   └─ 检查所有数字、日期、金额、工期一致性

9. AI 审查与预评分
   └─ 废标项检查、评分点覆盖、一致性检查、暗标检查、格式检查
   └─ 输出：审查报告 + 风险关闭清单 + 预评分报告

10. 人工终审与责任签发
    └─ 商务审报价/条款、技术审方案/偏离、法务审废标/合同、排版审格式
    └─ 高风险项必须责任人签字确认

11. 排版导出与归档复盘
    └─ Word/PDF 导出、签章、暗标清理、电子平台检查
    └─ 归档：最终版、证据链、响应矩阵、审查记录、评分反馈
    └─ 回流：高质量章节、失分原因、模板改进项入知识库
```

### 3.2 LLM 任务拆分（8 类核心任务）

| 任务 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| `classify_section` | chunk 文本 + 布局信息 | 条款类型标签 | 允许多标签，按重要性排序 |
| `extract_project_info` | 公告/封面/前附表 | 项目基础信息 JSON | 金额保留原文，不换算 |
| `extract_eligibility` | 资格审查章节 | 资质要求列表 | 识别 AND/OR/NOT 逻辑 |
| `extract_evaluation` | 评分表/评标办法 | 评分项列表 | 价格分只抽公式，不计算 |
| `extract_mandatory_clauses` | 投标人须知/评标办法 | 废标/否决条款列表 | 高召回优先，全部标记人工复核 |
| `extract_blind_bid_rules` | 暗标章节/格式要求 | 暗标规则包 | 未提到暗标时输出 false，不编造 |
| `build_response_matrix` | 已校验 requirements + 目录模板 + 企业资料匹配 | 响应矩阵 | 不得新增原文不存在的要求 |
| `generate_bid_section` | 章节信息 + 招标要求 + 授权资料 | 章节草稿 + 覆盖率检查 | 缺料不编造，必须引用 evidence_id |

## 四、关键技术设计

### 4.1 文档解析层

#### 4.1.1 解析策略路由

| 输入类型 | 优先策略 | 原因 |
|----------|----------|------|
| Word / DOCX | 原生解析 | 保留标题、样式、表格、批注、页眉页脚 |
| Excel / 清单 | 原生解析 | 保留 sheet、公式、单元格、合并单元格 |
| 可复制 PDF | PDF 文本层 + layout | 避免 OCR 错字，保留原文 |
| 扫描 PDF / 图片 | OCR + layout | 无文本层，只能识别 |
| 图纸 / 图片化表格 | 多模态 OCR + 人工复核 | 高风险，需保留图像证据 |

#### 4.1.2 多表示保真存储

同时保留多种表示，而非仅 Markdown：

- `raw_file`：原始文件
- `page_image`：每页图片
- `layout_json`：块、bbox、页码、阅读顺序
- `markdown_view`：给 LLM 快速阅读的视图
- `html_tables`：复杂表格保留 HTML
- `table_json`：结构化表格
- `text_chunks`：按章节/语义分块
- `evidence_index`：原文位置索引

### 4.2 LLM 抽取层

#### 4.2.1 提示词设计原则

采用 XML 标签分区结构：

```xml
<role>
你是投标文件解析与响应矩阵专家...
</role>

<task>
执行任务：extract_evaluation
只处理本次输入的章节/表格...
</task>

<rules>
1. 只抽取输入文本中明确出现的信息
2. 不得猜测、不得补全、不得根据常识编造
3. 每个结论必须带 evidence_quote 和 source
4. 缺失字段使用 null 或 not_found
5. 高风险字段标记 needs_human_review=true
</rules>

<schema>
{{JSON_SCHEMA}}
</schema>

<input>
{{MARKDOWN_TEXT + TABLE_HTML + LAYOUT_REFS}}
</input>

<output>
只输出符合 schema 的 JSON...
</output>
```

#### 4.2.2 结构化输出

优先级：

```
Provider Structured Outputs / Tool Calling
  > JSON mode + Pydantic/Zod 校验重试
  > Prompt-only JSON
```

每个抽取对象必须包含：
- `value`：字段值
- `evidence_quote`：原文摘录
- `source`：文件、页码、章节、block_id
- `confidence`：high/medium/low
- `needs_human_review`：布尔值
- `review_reason`：复核原因

### 4.3 规则引擎层

#### 4.3.1 硬规则（代码校验）

- **日期校验**：投标截止 ≤ 开标时间；保证金截止 ≤ 投标截止
- **金额校验**：最高限价、预算、报价、保证金单位一致
- **分值校验**：评分项合计 = 总分；价格分公式变量完整
- **资质校验**：等级、专业、有效期、主体一致性
- **人员校验**：证书、社保、劳动合同、项目角色匹配
- **暗标校验**：公司名、logo、人员姓名、文件属性、图片水印
- **版本校验**：补遗/澄清是否覆盖旧条款

#### 4.3.2 软规则（LLM 辅助）

- 逻辑完整性
- 语言风格一致性
- 评分倾向分析
- 风险提示

### 4.4 RAG 与企业资料层

#### 4.4.1 检索策略

```
结构化硬过滤（权限/有效期/密级）
  -> BM25/关键词召回
  -> 向量召回
  -> Rerank
  -> 证据片段返回
  -> LLM 基于证据生成
```

#### 4.4.2 资料治理

- **版本管理**：资料变更历史、版本对比、回滚机制
- **有效期管理**：到期提醒（提前 30/60/90 天）、自动标记过期
- **权限控制**：RBAC、资料分级（公开/内部/机密）、审计日志
- **质量检查**：过期、低分、无权限、错误资料拦截

### 4.5 生成与审查层

#### 4.5.1 分章节生成

每章生成输入必须包含：
- 章节目标
- 相关招标要求（requirement_id 列表）
- 允许引用的企业资料（asset_id 列表）
- 禁止事项（不得编造、不得承诺超范围内容）
- 风格和模板要求
- 输出格式

生成输出必须包含：
- `section_markdown`：章节正文
- `coverage_check`：已覆盖/未覆盖 requirement_id 列表
- `evidence_refs`：引用的 evidence_id 列表
- `unsupported_claims`：无证据的主张列表
- `needs_human_review`：是否需要人工复核

#### 4.5.2 审查规则

| 审查类型 | 检查内容 | 处理方式 |
|----------|----------|----------|
| 覆盖率检查 | 每个 requirement_id 是否被响应 | 标记缺失，进入补写队列 |
| 事实一致性 | 金额、日期、人员、项目全书一致 | 标记冲突，进入人工复核 |
| 废标项检查 | 是否触发废标条件 | 标记 fatal，必须人工确认 |
| 暗标检查 | 身份信息泄露风险 | 自动清理 + 人工抽检 |
| 格式检查 | 字体、字号、页码、签章、目录 | 自动修复 + 人工确认 |
| 评分点覆盖 | 每个评分项是否有对应内容 | 标记缺失，进入补写队列 |

## 五、安全与治理设计

### 5.1 数据安全

- **私有化部署**：模型、向量库、文档、日志全链路不出域
- **权限隔离**：项目级数据隔离，禁止跨项目访问
- **密级控制**：资料出域控制、模型供应商留存策略
- **审计日志**：覆盖上传、解析、检索、生成、导出、删除、权限变更

### 5.2 人机边界

| 环节 | AI 职责 | 人工职责 |
|------|---------|----------|
| 招标文件解析 | 抽取、分类、建议 | 确认关键条款、时间节点 |
| 响应矩阵生成 | 映射、建议、风险标注 | 确认责任人、关闭风险 |
| 章节生成 | 初稿、引用、覆盖率检查 | 审核内容真实性、技术可行性 |
| 报价/工期 | 提取公式、计算草稿 | 最终报价决策、工期承诺 |
| 废标/合规 | 召回风险、标注等级 | 最终裁决、法律风险确认 |
| 最终提交 | 格式检查、导出 | 签字盖章、提交责任 |

### 5.3 风险防控

| 风险 | 防控措施 |
|------|----------|
| LLM 幻觉 | 强制证据链、规则校验、人工复核 |
| OCR 错误 | 多表示保真、人工抽检、置信度标注 |
| 检索质量差 | 混合检索、重排序、评估体系 |
| 版本混乱 | 版本管理、有效期提醒、变更追踪 |
| 数据泄露 | 权限控制、加密存储、审计日志 |
| 暗标泄露 | 暗标检查、元数据清理、水印检测 |

## 六、演进路线

### 6.1 P0：最小可行产品（MVP）

目标：验证"招标文件解析 + 响应矩阵"

范围：
- 支持 PDF/Word 上传
- 原生解析优先，扫描件 OCR
- 输出 Markdown + 表格 HTML + 页码定位
- 抽项目基础信息、时间节点、资格、评分、废标、材料
- 输出 JSON 和 Excel 响应矩阵
- 支持人工修改和确认

验收指标：
- 项目基础信息字段准确率 ≥ 85%
- 资格/废标/评分项召回率 ≥ 90%
- 表格结构保真率 ≥ 80%
- 每条抽取是否有页码证据：100%

### 6.2 P1：投标文件生成版

目标：从响应矩阵生成目录和章节初稿

范围：
- 企业资料库：资质、人员、业绩、产品、历史方案
- 资料按章节授权引用
- 生成目录、技术方案、商务响应、偏离表、材料清单
- 检查评分覆盖、废标、资质、暗标、格式

底线：
- 不允许无证据生成资质、人员、业绩和承诺
- 所有高风险字段必须人工确认

### 6.3 P2：企业级生产版

目标：支持多项目协作和正式交付

范围：
- 多用户权限、版本管理、规则包配置
- Word/PDF 模板导出
- 签章、页码、附件顺序、暗标元数据检查
- 审计日志和复盘回流

### 6.4 P3：智能优化版

目标：提高质量和可复用性

范围：
- 评测集建设
- OCR/解析模型对比、chunk 策略 A/B
- RAG 召回与 rerank 评估
- 风险召回率评估

## 七、与通用复杂文档平台的关系

投标助手是通用复杂文档 AI 平台的**第一个垂直场景验证**。平台化抽象包括：

| 标书专用能力 | 通用平台抽象 | 说明 |
|-------------|-------------|------|
| 招标文件解析 | 源需求解析 | 把外部要求抽成结构化对象 |
| 评分点/废标项 | 要求与硬约束 | 任意复杂文档的"必须覆盖/不能违反" |
| 响应矩阵 | 要求-响应矩阵 | 防漏项的核心工件 |
| 企业知识库 | 资料资产库 | 可检索、可授权、可追溯 |
| 分章节生成 | 章节/组件级生成 | 长文档按章节生成、审查、确认 |

## 八、核心结论

1. **投标助手的本质是"文档工厂"，不是"写作助手"**。它需要先解析、再规划、后生成，以响应矩阵为核心驱动。

2. **"技能即提示词注入"合理但不够**。必须与模板、RAG、结构化中间产物、自适应技能选择和人工复核组合，才能达到专业水准。

3. **分章节生成是底线，一键整本生成是红线**。高价值项目必须逐章生成、逐项核验、人工终审。

4. **证据链是审计能力的基础**。每个字段、每句正文都必须能回跳到原文位置。

5. **规则引擎是防幻觉的最后一道防线**。日期、金额、分值、资质有效期等硬校验不能交给 LLM。

6. **人机边界必须清晰**。AI 是助手，不是投标负责人、法务、报价负责人或评委。

7. **优化目标应是"可用初稿率"，不是生成速度**。5 分钟出稿改 8 小时，不如 30 分钟出稿只需轻改。
