# 投标AI助手招标文件解析与评审要素抽取调研

## 调研问题

基于已有《通用复杂文档 AI 平台能力抽象与产品架构调研》，继续调研一个标准的投标 AI 助手应如何解析招标文件，如何抽取评分标准、暗标要求、废标项、采购需求、企业资质材料清单；是否需要 OCR；OCR 之后是否仍需要 AI；如果需要 AI，应如何设计，是否主要是提示词设计。

## 核心结论

1. 标准投标 AI 助手的入口不是“写标书”，而是“读标解析”。标准链路应先把招标公告、招标文件、补遗、澄清、附件、图纸、清单和电子投标平台说明转成可检索、可定位、可校验的结构化资料，再生成招标文件解析报告、响应矩阵、风险清单和材料清单；已有标书报告和 TextIn/Coze 实践都支持“先解析，再抽取，再风控，再建议”的流程。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md][来源: https://cloud.tencent.com/developer/article/2631225] 可信度：高。
2. OCR 不是可选项，而是复杂文件的基础设施之一。对于扫描件、图片化 PDF、盖章页、截图、复杂表格、图纸和跨页表格，必须使用 OCR、版面分析、表格解析和页码定位；但电子 Word、可复制文本 PDF、结构化 Excel 或平台 API 数据不应无差别 OCR，应优先保留原生文本、样式层级和表格结构。[来源: https://cloud.tencent.com/developer/article/2631225][来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction][来源: https://idp.jxdinfo.com] 可信度：高。
3. OCR 后仍然需要 AI/LLM。OCR 解决“看见文字”，但不能稳定解决否定条件、组合资质、评分公式、跨章节引用、废标触发、暗标泄露、材料归属和投标策略。LLM 应承担语义抽取、规则解释、风险归类、缺失信息追问和建议生成；评分公式、日期金额、权重合计、资质有效期等则应交给规则/代码校验。[来源: https://gitcode.csdn.net/6a294e5410ee7a33f27a8aee.html][来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction] 可信度：高。
4. 系统设计不能主要依赖提示词。提示词只是抽取层的一部分；真正可交付的架构至少包含文档解析/OCR、文档类型 schema、字段定义、结构化输出、规则库、企业资料库/RAG、响应矩阵、证据链、校验器、人工复核和审计日志。OpenAI Structured Outputs、Google Document AI 字段命名建议、智谱数据提取方案和 ExtractBench 都指向同一结论：结构正确、字段完整、缺失/null 可区分、逐字段评测，比“写一个长 prompt”更关键。[来源: raw/articles/2026-06-15-general-complex-document-ai-platform-architecture-research.md][来源: https://platform.openai.com/docs/guides/structured-outputs][来源: https://docs.cloud.google.com/document-ai/docs/label-documents][来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction][来源: https://arxiv.org/html/2602.12247] 可信度：高。
5. 评分标准、废标项和资质要求必须作为“一等对象”进入响应矩阵，而不是只写入摘要。每条评分项、资格项、实质性响应条款、暗标条款和材料要求都应有原文位置、结构化字段、响应章节、企业证据、责任人、状态、风险等级和人工确认记录。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md][来源: https://cloud.tencent.com/developer/article/2631225][来源: https://gitcode.csdn.net/6a294e5410ee7a33f27a8aee.html] 可信度：高。
6. 暗标要求应独立建模为“身份信息泄露规则包”。它不仅检查正文公司名，还要检查页眉页脚、作者元数据、图片水印、案例名称、资质证书、文件属性、签章痕迹、目录标题、附件命名和电子投标文件元数据。现有公开来源多把暗标作为风险/格式项提及，缺少高质量细则清单，因此具体规则需按地区、行业和招标文件原文验证。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md][来源: https://market.aliyun.com/products/201204006/cmgj00074082.html] 可信度：中。
7. 质量控制应采用“机器高召回 + 人工终裁”的模式。系统要尽量召回废标、资格、评分、材料、格式和暗标风险，但最终是否投标、是否响应、是否偏离、是否承诺、是否通过资质审查，必须由商务、技术、法务/合规和投标负责人确认。[来源: https://www.almawave.com/public-tenders-artificial-intelligence][来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md] 可信度：高。

## 内容整理

### 一、标准招标文件解析流程

#### 1. 文件收集与版本归集

输入文件应覆盖招标公告、正式招标文件、投标人须知、评标办法、合同条款、技术规范、采购需求、工程量清单、图纸、附件、补遗、澄清答疑、电子投标平台说明、CA/签章/加密说明。系统应保留原始文件，不覆盖原文，并建立版本表，记录文件名称、来源、发布时间、生效关系和优先级。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md]

必须特别处理补遗、澄清、答疑与原招标文件的冲突。标准做法是把它们进入同一“招标文件包”，在解析时为每条要求记录 `source-file`、`version`、`effective-date`、`supersedes` 和 `original-location`。该版本字段为本文综合设计，需在真实项目中验证。

#### 2. 文档解析与版面还原

解析层应输出 Markdown/HTML、章节树、表格结构、图片/印章/签名、页码、段落块、坐标或原文定位。TextIn xParse 实践强调，招标文件常见 200-500 页、复杂表格、附件、扫描页和版式结构；先转成 Markdown + 结构化信息，再交给 LLM 抽取和风控，更稳定。[来源: https://cloud.tencent.com/developer/article/2631225]

金现代 IDP 来源也显示，文档解析应支持 PDF、Word、Excel 等格式，结合大语言模型和 OCR 提取文字、表格、公式、图片；招投标全流程评审需要检查公告与招标文件一致性、文件发售期、费用合规、评标公告关键信息一致性、资质证明验证等。[来源: https://idp.jxdinfo.com]

#### 3. 章节与条款切分

系统应把招标文件切分为：项目基础信息、投标人须知、资格审查、符合性审查、评标办法、采购需求、技术规范、商务条款、合同条款、投标文件格式、附件清单、电子提交规则、暗标规则、废标/否决条款、澄清补遗。切分不是简单按标题分段，还要保留跨页表格、条款引用和附件归属。

招标文件解析难点包括非结构化版式、长文档稀疏信息、否定/条件逻辑、跨页表格和跨章节引用。[来源: https://gitcode.csdn.net/6a294e5410ee7a33f27a8aee.html]

#### 4. 结构化抽取

建议将抽取分为 10 类对象：

- `ProjectInfo`：项目名称、编号、标段、采购人、代理机构、预算/最高限价、采购方式。
- `Timeline`：报名、答疑、澄清、保证金、投标截止、开标、评审、交付节点。
- `EligibilityRequirement`：主体资格、资质证书、业绩、人员、财务、信誉、联合体、分包、授权。
- `EvaluationCriteria`：技术分、商务分、价格分、客观分、主观分、权重、公式、得分证据。
- `MandatoryClause`：实质性响应、否决投标、无效投标、废标项、一票否决。
- `ProcurementRequirement`：采购内容、数量、范围、技术参数、服务标准、交付成果、验收标准。
- `BidDocumentRequirement`：文件组成、章节顺序、格式、签章、装订、页码、报价表、偏离表。
- `BlindBidRequirement`：暗标范围、禁止标识、版式统一、匿名化、元数据清理。
- `MaterialRequirement`：营业执照、资质证书、人员证书、业绩证明、财务报表、信用证明、授权文件。
- `RiskAndAction`：风险等级、触发原文、建议动作、责任人、关闭状态。

智谱招投标数据提取方案给出的字段定义、默认值、异常处理和 JSON 输出思路，可作为基础字段抽取模板；但投标助手应扩展到评分、废标、暗标和响应矩阵。[来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction]

#### 5. 响应矩阵生成

响应矩阵是投标 AI 助手的核心中间产物。每条要求都应映射到投标文件章节、企业资料证据、负责人、完成状态、风险等级、是否必须响应、是否需人工确认。已有标书报告指出，没有响应矩阵，AI 生成内容容易变成看似完整但无法证明逐项响应的通用方案。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md]

推荐字段：

- 要求 ID：稳定引用编号，如 `REQ-QUAL-001`、`REQ-SCORE-003`。
- 要求类型：资格、评分、废标、采购需求、格式、暗标、材料、时间、合同。
- 原文摘录：保留关键条款原文。
- 原文位置：文件、页码、章节、段落、表格单元格。
- 响应方式：直接响应、偏离说明、提供证明、人工决策、不可响应。
- 响应章节：投标文件目录中的具体章节。
- 证据材料：企业资料库文件、资质证书、人员证书、业绩合同、授权书等。
- 责任人：商务、技术、法务、财务、资质管理员、项目经理。
- 风险等级：高、中、低。
- 关闭状态：未处理、已匹配、待补资料、待澄清、已确认、接受风险。

### 二、OCR 适用边界

OCR 应用于以下场景：

- 扫描版 PDF、图片、截图、拍照文件。
- 盖章页、签字页、资质证书、人员证书、授权书。
- PDF 文字层缺失、乱码、顺序错乱或不可复制。
- 图纸、流程图、图片化技术参数。
- 复杂表格、合并单元格、跨页表格需要结合版面/表格解析。

OCR 不应无差别替代原生解析：

- Word、Excel、可复制文本 PDF 应优先保留原始文本、标题层级和表格对象。
- OCR 可能丢失上标、公式、表格结构、页眉页脚语义、大小写和符号。
- 对报价、金额、日期、证书编号、项目编号等高风险字段，应保留原文截图/页码，并做人工确认。

TextIn/Coze 实践指出，大模型直接硬读全文常因长文档截断、扫描件/复杂表格解析不可靠和缺少原文引用而失败；更稳妥做法是先把文档解析为结构化可理解内容，再让 LLM 做抽取和风险识别。[来源: https://cloud.tencent.com/developer/article/2631225]

### 三、评分标准抽取

评分标准需要从自然语言、表格和公式中抽取为可计算对象。最低字段包括：

- 评分办法名称：综合评分法、最低评标价法、低价优先法、平均价法等。
- 分值结构：技术分、商务分、价格分、服务分、资信分、客观分、主观分。
- 每项评分点：名称、分值、评分描述、得分条件、扣分条件、证明材料。
- 权重和满分：分项分值、总分、是否加总为 100 或招标文件规定满分。
- 公式：价格分、商务分、技术指标响应分等的可执行表达式。
- 依赖字段：报价、基准价、有效投标人数量、最高限价、平均价、下浮率等。
- 原文位置：评分表页码、章节、表格行列。
- 响应章节：每个评分点应落到投标文件章节或附件。

招标 Agent 来源提出将评分细则从自然语言转为计算公式：识别规则类型，将公式转为 Python 可执行代码，生成评分模拟函数，并检查技术分、商务分、价格分权重之和是否为 100%。[来源: https://gitcode.csdn.net/6a294e5410ee7a33f27a8aee.html]

系统设计建议：

1. 用文档解析层保留评分表结构，不把表格压扁成普通段落。
2. 用 LLM 抽取评分项、分值、条件、证明材料和原文引用。
3. 用规则/代码校验分值合计、公式变量、金额单位、百分比和边界条件。
4. 对主观评分项输出“响应写作要点”，对客观评分项输出“证据材料需求”。
5. 对价格分公式生成可审计脚本，但不得让 AI 单独给出报价决策。

### 四、暗标要求识别

暗标识别应从“是否有暗标要求”升级为“暗标规则包 + 匿名化检查器”。标准抽取字段包括：

- 是否暗标：明标、暗标、技术暗标、商务明标/技术暗标混合。
- 暗标范围：技术标、施工组织设计、服务方案、图纸、附件、电子文件。
- 禁止信息：公司名称、简称、logo、商标、地址、电话、联系人、人员姓名、项目案例可识别信息。
- 禁止载体：正文、页眉页脚、封面、目录、图片、水印、表格、证书、附件名、文件名、文档属性、批注、修订记录。
- 格式统一要求：字体、字号、行距、页边距、页码、标题编号、颜色、图表样式。
- 处理动作：匿名化替换、删除元数据、统一样式、图片脱敏、案例泛化、人工确认。
- 交付校验：Word/PDF 元数据检查、图片 OCR 复检、文件名检查、压缩包目录检查。

已有标书报告明确将暗标泄露身份列为高风险坑点：作者信息、页眉页脚、图片水印、案例公司名和文件元数据都可能暴露企业身份。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md]

暗标规则的公开权威资料在本次检索中不足。建议产品内置“通用暗标检查项”，但每个项目必须以招标文件原文为准，由投标负责人确认具体规则。

### 五、废标项与实质性响应条款识别

废标项应作为最高风险对象处理。标准抽取范围包括：

- 明示废标：出现“废标”“否决投标”“无效投标”“不予受理”“一票否决”等表述。
- 实质性响应：资格、技术参数、商务条款、报价、工期、质量、安全、服务承诺。
- 文件格式：签章、授权、法定代表人证明、保证金、投标函、报价表、偏离表。
- 时间与提交：逾期、上传失败、文件命名、加密、CA、递交平台。
- 报价风险：超过最高限价、低于成本价、不平衡报价、漏项、大小写不一致。
- 资质风险：证书缺失、过期、等级不满足、人员社保/证书不匹配。
- 暗标风险：身份泄露、格式不统一、元数据未清理。

风险识别输出应至少包含：风险点、严重等级、触发原文、风险类型、影响后果、建议动作、责任人、是否已关闭。TextIn/Coze 实践中的风险 prompt 包含风险点、触发原文、风险类型和建议动作，适合扩展为正式风险清单 schema。[来源: https://cloud.tencent.com/developer/article/2631225]

### 六、采购需求结构化

采购需求应从“需求摘要”拆成“可响应对象”。建议字段：

- 标的与范围：货物、服务、工程、软件、设备、施工、运维、培训等。
- 数量与规格：数量、单位、规格型号、关键参数、兼容性、品牌限制或等同要求。
- 技术要求：功能、性能、接口、安全、可靠性、验收指标、标准规范。
- 商务要求：交付期、质保期、付款、履约保证、售后、培训、响应时间。
- 服务要求：人员配置、项目组织、驻场、巡检、应急、运维等级。
- 验收要求：交付物、验收流程、验收标准、测试报告、第三方证明。
- 约束条件：国产化、信创、安全、环保、地域、资质、业绩、行业规范。
- 响应方式：逐条响应、偏离表、技术证明、产品彩页、检测报告、截图、承诺函。

采购需求常分散在技术规范、合同条款、附件、清单和评分办法中。系统应建立需求项与评分项、废标项、材料项之间的关联，避免只在“采购需求”章节抽一次就结束。

### 七、企业资质材料清单抽取

资质材料清单应从招标要求和企业资料库两侧生成。

招标侧抽取字段：

- 企业主体材料：营业执照、法人证明、授权委托书、开户许可证或基本账户证明。
- 资质证书：行业资质、等级、专业、有效期、安全生产许可证、认证证书。
- 人员材料：项目经理、技术负责人、专职人员、证书编号、社保、劳动合同、简历。
- 业绩材料：合同、中标通知书、验收报告、金额、时间、项目类型、客户证明。
- 财务材料：财报、审计报告、纳税证明、社保证明、银行资信。
- 信用材料：信用中国、政府采购严重违法失信记录、无重大违法声明。
- 产品/技术材料：检测报告、授权书、厂家证明、产品彩页、参数响应证明。
- 其他材料：保证金凭证、承诺函、联合体协议、分包协议、保险、售后网点证明。

企业侧匹配字段：

- 资料文件、版本、有效期、适用范围、授权状态、密级、来源、最近复核时间。
- 与招标要求的匹配结论：满足、不满足、存疑、需更新、需人工确认。
- 证据位置：证书页码、编号、关键字段截图或 OCR 结果。

招标 Agent 来源提出把资格要求解析为资质树：识别资质实体，抽取 AND/OR/排除逻辑，再与企业资质库比对，输出满足、不满足或存疑。[来源: https://gitcode.csdn.net/6a294e5410ee7a33f27a8aee.html]

## AI/LLM 介入点

LLM 应介入但不应独立裁决的环节：

1. 条款分类：判断一段文字属于资格、评分、废标、采购需求、合同、格式、暗标还是材料要求。
2. 语义抽取：抽取否定条件、组合条件、替代条件、证明材料、风险后果。
3. 评分解释：把自然语言评分项转成字段、公式草稿和响应写作要点。
4. 风险识别：结合原文和企业资料，输出高/中/低风险和建议动作。
5. 缺失信息追问：识别企业资料库中缺少的资质、业绩、人员或证明。
6. 响应建议：为每条要求建议响应章节、证据材料和偏离说明。
7. 章节生成：在响应矩阵和资料授权边界内生成初稿。
8. 审查说明：把校验结果转成可读的整改建议。

不应由 LLM 单独完成的环节：

- OCR、版面/表格解析的底层识别。
- 日期、金额、权重、公式和字段完整性的硬校验。
- 证书真伪、资质有效期和信用记录的最终确认。
- 报价策略、是否投标、是否承诺、是否接受偏离的商业决策。
- 法律风险、废标风险和合同风险的最终裁决。

## 提示词与结构化输出设计

### 一、提示词不是主架构

提示词负责“让模型知道怎么抽”，但产品稳定性来自“schema + 解析质量 + 规则 + 校验 + 复核”。智谱方案强调字段定义、异常处理、结构化输出和数据校验；Google Document AI 标注文档强调字段命名会影响抽取准确性，应使用与文档相同语言且避免缩写；OpenAI Structured Outputs 说明 JSON Schema 可提高输出结构一致性，但结构正确不等于事实正确。[来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction][来源: https://docs.cloud.google.com/document-ai/docs/label-documents][来源: https://platform.openai.com/docs/guides/structured-outputs]

### 二、推荐 schema

建议每个抽取对象包含以下通用字段：

```json
{
  "id": "REQ-QUAL-001",
  "type": "eligibility_requirement",
  "title": "建筑工程施工总承包二级资质",
  "original_text": "投标人须具备建筑工程施工总承包二级及以上资质...",
  "source": {
    "file": "招标文件.pdf",
    "page": 12,
    "section": "第二章 投标人须知",
    "block_id": "p12-b03"
  },
  "logic": {
    "operator": "AND",
    "conditions": []
  },
  "is_mandatory": true,
  "risk_level": "high",
  "response_target": {
    "bid_section": "资格证明文件",
    "materials": []
  },
  "confidence": "medium",
  "needs_human_review": true
}
```

### 三、提示词设计原则

- 明确输入边界：用标签包裹解析后的章节、表格、页码和原文片段。
- 明确字段定义：每个字段说明含义、类型、取值范围、缺失值处理。
- 强制证据输出：每条结论必须带原文摘录和位置。
- 区分事实与推断：抽取事实、风险判断、建议动作分开输出。
- 缺失不编造：没有原文证据时输出“未找到/需人工确认”。
- 分任务提示：先抽项目基础信息，再抽资格，再抽评分，再抽废标，不要一次性抽全篇。
- 输出可校验 JSON：所有字段进入 JSON Schema 或 Pydantic/Zod 校验。
- 对高风险字段二次确认：资质、金额、日期、评分公式、废标、暗标必须复核。

### 四、示例任务拆分

1. `extract_project_info`：抽项目名称、编号、采购人、预算、时间节点。
2. `extract_eligibility`：抽主体资格、资质、业绩、人员、财务、信誉、联合体。
3. `extract_evaluation`：抽评分办法、分值、公式、证明材料、响应建议。
4. `extract_mandatory_clauses`：抽废标、否决投标、实质性响应、一票否决。
5. `extract_procurement_needs`：抽采购范围、技术参数、交付和验收。
6. `extract_blind_bid_rules`：抽暗标要求、身份泄露载体和匿名化动作。
7. `build_response_matrix`：汇总要求、章节、证据、负责人和风险状态。
8. `review_against_rules`：用规则库和企业资料库做校验。

## 质量控制、校验与人工复核

### 机器校验

- JSON 校验：字段是否存在、类型是否正确、枚举是否合法。
- 日期校验：投标截止时间不得晚于开标时间；保证金截止不得晚于投标截止。
- 金额校验：最高限价、预算、报价、保证金、履约保证金单位和大小写一致。
- 分值校验：评分项合计、权重、公式变量是否完整。
- 资质校验：等级、专业、有效期、人员证书、社保、业绩时间是否匹配。
- 材料完整性校验：每条资格/评分/格式要求是否有对应材料或响应章节。
- 暗标校验：公司名、logo、人员姓名、文件属性、图片水印、页眉页脚、附件名。
- 版本校验：补遗/澄清是否覆盖旧条款，响应矩阵是否已更新。
- 溯源校验：每条风险、字段和建议是否有原文位置。

智谱方案提供了格式校验、逻辑校验、完整性校验和一致性校验的通用思路，例如日期格式、金额格式、投标截止时间与开标时间顺序、项目编号一致性等。[来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction]

### 人工复核

人工复核应按责任分工：

- 商务负责人：资格、报价、保证金、商务条款、提交规则。
- 技术负责人：采购需求、技术参数、实施范围、评分响应、偏离。
- 法务/合规：废标、合同、授权、联合体、分包、承诺边界。
- 资质管理员：企业证照、人员证书、业绩证明、有效期。
- 排版/投标专员：格式、签章、页码、附件顺序、暗标、电子平台上传。

Almawave 公共招标 AI 文章明确指出，生成式 AI 不能替代起草招标文件所需的法律和技术专业知识，应作为支持工具贯穿流程。[来源: https://www.almawave.com/public-tenders-artificial-intelligence]

## 推荐系统架构

```text
项目工作区
  ├─ 文件包管理：公告 / 招标文件 / 补遗 / 澄清 / 附件 / 图纸 / 平台说明
  ├─ 解析层：原生解析 / OCR / Layout / 表格 / 图片 / 页码定位 / 版本差异
  ├─ 抽取层：项目字段 / 资格 / 评分 / 废标 / 暗标 / 采购需求 / 材料清单
  ├─ 规则层：废标规则 / 评分校验 / 资质匹配 / 暗标检查 / 格式规则
  ├─ 企业资料层：资质库 / 人员库 / 业绩库 / 产品库 / 历史标书 / 授权与有效期
  ├─ 响应矩阵：要求 → 章节 → 资料 → 责任人 → 风险 → 复核状态
  ├─ 生成层：目录 / 章节草稿 / 偏离表 / 材料清单 / 澄清问题 / 风险报告
  ├─ 审查层：覆盖率 / 一致性 / 原文溯源 / 预评分 / 人工终审
  └─ 交付层：Word / PDF / 附件包 / 签章 / 暗标清理 / 提交检查单 / 归档复盘
```

该架构延续已有通用复杂文档平台报告中的“文档类型配置 + 资料资产 + 需求解析 + 规划编排 + 分章节生成 + 规则审查 + 证据溯源 + 排版交付”框架，并将标书场景细化为专用解析包、评分/废标规则包、暗标规则包、资质材料包和响应矩阵。[来源: raw/articles/2026-06-15-general-complex-document-ai-platform-architecture-research.md]

## 推荐工作流

1. 建立项目：录入项目名称、投标主体、负责人、截标时间、密级和权限。
2. 上传文件包：招标文件、附件、补遗、澄清、图纸、清单和平台说明。
3. 文档解析：原生解析优先，扫描/图片/复杂版式使用 OCR 和版面分析。
4. 章节切分：生成章节树、表格结构、页码定位和版本差异。
5. 字段抽取：抽项目基础信息、时间节点、资格、评分、废标、采购需求、暗标、材料。
6. 生成解析报告：所有关键结论附原文页码、片段和置信度。
7. 建响应矩阵：把要求映射到投标章节、企业资料、责任人和风险状态。
8. 企业资料匹配：从资质库、人员库、业绩库、产品库匹配证明材料。
9. 生成缺失清单：输出缺资质、缺业绩、缺授权、缺证明、需澄清问题。
10. 分章节生成：在响应矩阵和授权资料范围内生成章节初稿。
11. AI 审查：检查评分覆盖、废标、资质、暗标、格式、报价/工期/人员一致性。
12. 人工终审：商务、技术、法务、资质和排版按责任域确认。
13. 导出交付：生成 Word/PDF、附件包、签章文件和提交检查单。
14. 归档复盘：保存最终版、风险关闭清单、评分反馈和知识库更新项。

## 适用场景

- 企业高频参与政府采购、工程、设备、IT 服务、运维服务、集成项目等投标。
- 招标文件页数多、附件多、评分标准复杂、废标风险高。
- 企业已有资质、人员、业绩、产品资料和历史标书，可建设资料库。
- 项目需要多部门协作、版本审计、提交检查和复盘沉淀。
- 涉密或高价值项目，但前提是支持私有化/混合部署、权限和审计。

## 不适用场景

- 招标文件极短、要求简单、低频低金额项目，人工处理成本更低。
- 企业没有真实资料库，期望 AI 编造资质、业绩、人员或承诺。
- 招标文件扫描质量差到 OCR 和人工都无法确认关键条款。
- 高度策略性报价、合同风险或法律责任问题，不能让 AI 单独决策。
- 组织不愿维护规则库、资料库、模板和人工复核流程。

## 风险与约束

1. OCR 漏识别：扫描件、跨页表格、复杂图纸、印章遮挡、低清晰度图片会导致漏项，需人工抽检。[来源: https://cloud.tencent.com/developer/article/2631225]
2. LLM 幻觉：模型可能补全不存在的资质、公式或风险，必须要求原文证据和缺失输出。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md]
3. 评分公式错误：公式抽取后必须由代码和人工校验，不得直接采信模型计算。
4. 废标项漏召回：废标条款分散在投标人须知、评标办法、格式、合同和电子平台说明中，需多轮抽取。
5. 暗标细则不足：公开来源缺少统一暗标规则，必须按项目原文配置。
6. 企业资料过期：证书、人员、业绩、授权、检测报告可能过期或不适用。
7. 供应商宣传指标不可直接采信：本次证据中多处效率、准确率、中标率、废标率数字来自厂商或推广文章，缺少第三方验证。
8. 数据安全：招标文件、报价、资质、客户策略和历史标书属于敏感资料，应控制模型、向量库、日志和文档出域。
9. 责任边界：AI 是助手，不是投标负责人、法务、报价负责人或评委。

## 信源质量门控记录

### 门控规则

- A 级：官方文档、学术论文、产品正式文档、云厂商/平台文档，可支撑核心事实。
- B 级：较可信的行业文章、厂商技术文章、开源项目资料，可支撑实践参考，但营销数字需验证。
- C 级：推广性强、单方宣传、财经/社区文章，只作背景或交叉验证，不单独支撑核心结论。
- D 级：低相关、重复、抓取失败、无法访问或明显广告，剔除。
- C/D 级来源不得作为唯一事实依据；供应商宣称的准确率、效率、中标率、废标率仅作为待验证信息。
- 入库映射：A/B → `source-quality: high`；C → `source-quality: medium`；D → `source-quality: low`。

## 对于可信度较高的来源逐来源总结

### 1. 通用复杂文档 AI 平台架构调研

正文内容：该报告将复杂文档平台抽象为“文档类型配置 + 资料资产 + 需求解析 + 规划编排 + 分章节生成 + 规则审查 + 证据溯源 + 排版交付”的文档工厂，并强调标书场景需要专用解析包、评分/废标规则包、企业资料包、表单附件包、输出格式包和评审包。[来源: raw/articles/2026-06-15-general-complex-document-ai-platform-architecture-research.md]

可用事实：本报告的架构设计应继承“文档类型 schema、响应矩阵、证据链、规则审查、人工复核”的原则。

局限：该报告偏平台抽象，本次需要进一步细化招标文件解析对象。

适合入库知识点：复杂文档 AI 平台、标书专用规则包、响应矩阵。

### 2. 国内 AI 标书助手能力、坑点与标准 SOP 调研

正文内容：该报告总结国内 AI 标书助手主流形态为“招标文件解析 + 企业知识库/RAG + 分章节生成 + 合规审查 + 排版导出”，并列出已实现功能：项目基础信息、资格要求、评分标准、废标项、投标文件格式、暗标、企业资料库、偏离表、资质证明清单、合规检查、预打分和 Word/PDF 输出。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md]

可用事实：投标助手必须以读标和响应矩阵为中心；一键生成整本标书只能作为低风险初稿能力。

局限：部分来源是国内厂商或推广文章，准确率和中标率等数字需要企业 POC 验证。

适合入库知识点：AI 标书助手 SOP、废标项检查、暗标风险、企业标书知识库。

### 3. TextIn xParse + Coze 招标文件解析智能体实践

正文内容：文章指出招标文件常见 200-500 页，时间节点密集，关键条款分散，风险提示必须追溯到原文位置。大模型直接硬读全文会因长文档截断、扫描件/复杂表格解析不可靠和缺少原文引用而失败。方案使用 TextIn xParse 将文件解析为 Markdown + 结构化信息，再用 Coze/LLM 抽取关键条款、风险和响应建议。[来源: https://cloud.tencent.com/developer/article/2631225]

可用事实：OCR/文档解析是 LLM 前置层；输出应包含关键条款摘要、风险提示和响应建议，且尽量带页码/引用。

局限：文章是厂商技术实践，不等同于第三方评测。

适合入库知识点：招标文件解析智能体、OCR + LLM 工作流、风险清单 prompt。

### 4. 智谱 AI 数据提取最佳实践

正文内容：官方文档以招投标为场景，说明 PDF、Word、网页、扫描件格式复杂，传统 OCR/规则在表格嵌套、金额大小写、字段缺失时效果不理想；提出输入、预处理、LLM prompt、JSON 输出、后处理、格式校验、逻辑校验、完整性校验、一致性校验和修复模块。[来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction]

可用事实：投标助手抽取层应以字段定义、异常处理、结构化输出、JSON 修复、日期/金额/逻辑校验为工程基础。

局限：页面示例更偏公告/中标信息抽取，未完整覆盖评分、暗标、废标。

适合入库知识点：招投标结构化抽取、LLM 数据提取 prompt、抽取后校验。

### 5. ExtractBench

正文内容：论文提出复杂结构化 PDF 抽取 benchmark，包含 35 个 PDF、12867 个可评测字段、5 个高价值领域，强调 schema-driven evaluation、逐字段指标、语义数组对齐、显式 missing/null 处理；前沿模型在企业级大 schema 上仍可能不可靠。[来源: https://arxiv.org/html/2602.12247]

可用事实：投标助手不应只看“整体摘要像不像”，而要逐字段评测评分项、资质项、废标项和材料项是否抽对。

局限：论文不是招投标专用，但对复杂结构化抽取可靠性非常相关。

适合入库知识点：复杂结构化抽取评测、schema-driven evaluation。

### 6. Almawave Public Tenders and AI

正文内容：文章讨论公共招标中的 AI 应用，指出 AI 可帮助起草、审查、合规检查、供应商资格评估、资料检索和投标文档生成；但生成式 AI 不能替代法律和技术专家，只能作为支持工具贯穿招标设计与管理流程。[来源: https://www.almawave.com/public-tenders-artificial-intelligence]

可用事实：投标助手应定位为人机协同工具，尤其在公共采购和合规场景中必须保留专家责任。

局限：文章含产品宣传和欧洲公共采购背景，需结合中国招投标规则验证。

适合入库知识点：公共招标 AI 助手、AI 合规审查、投标人资格评估。

### 7. 金现代智能文档处理平台

正文内容：产品页面列出文档解析、图纸解析、查重对比、规则库、招投标全流程评审员、资质证明验证、格式规范检测、审查报告导出等能力；文档解析支持 PDF、Word、Excel 等格式，并结合大语言模型和 OCR 提取文字、表格、公式、图片。[来源: https://idp.jxdinfo.com]

## 跨源矛盾检测结论

### 检测范围

- 已阅读本地核心 raw 报告 2 篇、脚本证据包精读来源 10 个。
- 检测对象：核心数字、日期、功能描述、因果判断、市场/公司事实。
- 检测时间：2026-06-16。

### 检测结果

结论：未发现会推翻“投标 AI 助手应采用 OCR/文档解析 + LLM 抽取 + 规则校验 + 响应矩阵 + 人工复核”的实质矛盾，但存在 5 类差异或待验证项。

1. 自动化程度差异：部分产品强调“一键生成”或“数分钟生成”，而本地标书调研、TextIn/Coze 实践和 Almawave 均支持更稳妥的人机协同流程。综合判断：一键生成可作为初稿或低风险标准化入口，不应作为正式投标交付唯一流程。[来源: raw/articles/2026-06-15-ai-bid-document-assistant-research.md][来源: https://cloud.tencent.com/developer/article/2631225][来源: https://www.almawave.com/public-tenders-artificial-intelligence]
2. OCR 与 LLM 角色边界差异：有些厂商页面把“AI 解析”整体包装为单一能力，但技术实践显示 OCR/版面解析和 LLM 语义抽取职责不同。综合判断：解析层和 LLM 层应解耦。[来源: https://cloud.tencent.com/developer/article/2631225][来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction]
3. 准确率/效率数字矛盾或不可验证：艾瑞数智、金现代、数商云等来源包含效率提升、识别率、废标率、准确率等数字，但多为官网或推广文案，缺少第三方评测；本报告不把这些数字作为事实结论。[来源: https://www.idigital.com.cn/common-ai-2][来源: https://idp.jxdinfo.com][来源: https://caifuhao.eastmoney.com/news/20260210100616089577970]
4. 结构化输出能力边界：OpenAI Structured Outputs 和智谱 prompt 示例能提高 JSON 结构稳定性，但 ExtractBench 显示企业级大 schema 下前沿模型仍可能不可靠。综合判断：schema 是必要条件，不是充分条件，必须做逐字段评测和人工复核。[来源: https://platform.openai.com/docs/guides/structured-outputs][来源: https://docs.bigmodel.cn/cn/best-practice/case/data-extraction][来源: https://arxiv.org/html/2602.12247]
5. 暗标规则证据不足：多份标书资料提到暗标风险，但本次外部检索没有找到高质量、通用、权威的暗标自动检查规则清单。综合判断：暗标应进入系统架构，但具体规则需以招标文件和地方/行业规则为准。

## 矛盾与待验证问题

1. 各产品实际废标项召回率、评分点覆盖率、暗标检查准确率、资质匹配准确率缺少公开第三方评测。
2. 暗标规则高度依赖项目原文、地区、行业和电子投标平台，通用规则包只能做基线，不能替代逐项目配置。
3. 评分公式自动转代码的可靠性需要用真实评分办法样本评测，尤其是平均价法、区间扣分、异常低价和多标段公式。
4. 企业资质库与外部平台证书真伪校验、信用记录校验的接口可用性和合法合规边界需单独验证。
5. OCR 对扫描件、图纸、跨页表格、盖章遮挡和低清晰度图片的漏项率需要按企业样本测试。
6. 私有化部署是否做到文档、模型、向量库、日志、审计、临时文件全链路不出域，需要技术审计。
7. 供应商宣传的效率提升、中标率提升、零废标等指标不能作为采购依据，应以企业自有 POC 指标验证。
