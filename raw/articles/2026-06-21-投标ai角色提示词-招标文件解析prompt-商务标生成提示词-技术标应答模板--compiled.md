# 投标 AI 角色提示词与招标文件解析 Prompt 调研报告

## 核心结论

1.  **AI 在投标中的定位是辅助而非替代**：AI 工具仅作为信息提取与比对预处理工具，不具备法定评标决策权，所有结论需人工复核，最终废标决定权归属评标委员会。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html] 可信度：高
2.  **招标文件解析需结合结构化提取与大模型**：仅靠大模型直接“硬读”全文常会翻车（文档过长截断、无法可靠解析扫描件/表格），更稳的打法是先把文档变成结构化可理解的内容（如 Markdown+ 结构化信息），再让大模型做抽取和风控。[来源：https://cloud.tencent.com/developer/article/2631225] 可信度：高
3.  **提示词工程需遵循特定框架与约束**：有效的提示词设计应包含角色设定、任务目标、输入数据、输出格式、约束条件（如“仅依据提供的文本客观比对，禁止推测”），推荐使用 CRISPE、CARE 等框架。[来源：https://adg.csdn.net/6973116c437a6b40336b7ea4.html] 可信度：中
4.  **私有知识库与 RAG 是提升专业性的关键**：通过将企业历史标书、资质人员信息、政策文件等结构化知识嵌入生成流程，可解决传统标书内容泛化、专业性不足痛点，实现从“通用问答”到“行业专属”跃迁。[来源：https://www.53ai.com/news/zhinenghuagaizao/2025022571652.html] 可信度：中
5.  **投标 workflow 需分阶段介入**：AI 最适合放在中段（搭目录、写章节、整理材料）和后段（润色、统一风格、检查表达），前段（读招标文件、拆评分点、判断响应策略）涉及投标策略和合规判断，要由人把关。[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html] 可信度：高
6.  **文档解析需处理复杂版式与溯源**：招标文件包含复杂表格、页眉页脚、附件、扫描页，解析能力需覆盖电子档 PDF、扫描件、图片，且输出需包含可追溯的原文定位信息（第几页、哪一段）。[来源：https://cloud.tencent.com/developer/article/2631225] 可信度：中
7.  **效率提升数据显著但需验证**：部分平台声称投标效率提升 70% 以上，初稿生成从数天缩短至 2 小时或 10 分钟，客户满意度达 98%，但具体数据依赖特定平台测试。[来源：https://www.kuaibiaobao.com] 可信度：低

### 1. 招标文件解析与数据提取技术

#### 1.1 文件内容提取代码实现
针对 Word 和 PDF 文件，需采用特定库进行内容提取，以确保表格和结构不被丢失。

**Word 文件提取（docx2python）：**
```python
content = None
try:
    # 使用 docx2python 提取，它能更好地处理表格和结构
    content = docx2python(file_path)
    extracted_text = []
    # 处理文档内容
    if hasattr(content, 'document'):
        for section in content.document:
            for element in section:
                if isinstance(element, list):
                    # 这可能是表格
                    extracted_text.append("\n[表格内容]")
                    for row in element:
                        if isinstance(row, list):
                            row_text = " | ".join([str(cell).strip() for cell in row if cell])
                            if row_text:
                                extracted_text.append(row_text)
                        else:
                            extracted_text.append(str(row))
                    extracted_text.append("[表格结束]\n")
                else:
                    # 普通文本
                    text = str(element).strip()
                    if text:
                        extracted_text.append(text)
    result = "\n".join(extracted_text).strip()
    # 确保释放资源
    if content:
        del content
    gc.collect()
    return result
except Exception as e:
    # 确保释放资源
    if content:
        del content
    gc.collect()
```
[来源：https://www.cnblogs.com/yibiaoai/p/19064673]

**PDF 文件提取（pdfplumber）：**
```python
pdf = None
try:
    extracted_text = []
    pdf = pdfplumber.open(file_path)
    for page_num, page in enumerate(pdf.pages, 1):
        # 添加页码标识
        extracted_text.append(f"\n--- 第 {page_num} 页 ---\n")
        # 提取普通文本
        text = page.extract_text()
        if text:
            extracted_text.append(text)
        # 提取表格
        tables = page.extract_tables()
        for table_num, table in enumerate(tables, 1):
            extracted_text.append(f"\n[表格 {table_num}]")
            for row in table:
                if row:  # 跳过空行
                    # 过滤空值并连接单元格
                    row_text = " | ".join([str(cell) if cell else "" for cell in row])
                    extracted_text.append(row_text)
            extracted_text.append("[表格结束]\n")

    result = "\n".join(extracted_text).strip()
    # 确保关闭 PDF 文件
    if pdf:
        pdf.close()
    gc.collect()
    return result
except Exception as e:
    # 确保关闭 PDF 文件
    if pdf:
        pdf.close()
    gc.collect()
```

#### 1.2 智能文档解析引擎优势
招标文件不是纯文本，是动辄几十页到几百页的长文档，而且是“文档版面理解”的集合：标题层级、目录结构、表格、页眉页脚、附件、扫描页。解析层决定“能不能用”。

*   **复杂版式 + 表格 + 长文档解析能力**：核心值是把复杂结构转换成更适合大模型处理的 Markdown+ 结构化信息，大幅降低“模型看不懂文档”的概率。
*   **输出可追溯的原文定位信息**：投标场景里，单单给出风险提示没用，必须能指回原文：第几页、哪一段、哪张表。输出通常包含 markdown（正文）+ 结构块信息 + 页级信息。
*   **覆盖更多真实文件形态**：电子档 PDF、扫描件、图片都能处理。解析能力覆盖越全，Demo 越不容易翻车。
[来源：https://cloud.tencent.com/developer/article/2631225]

#### 1.3 学术框架：半结构化招标文件检索增强生成
*   **目标**：开发一种方法，允许 LLM 从文本提示生成半结构化文档，同时保持原始内容的完整性。
*   **模块**：
    *   **Template Retrieval Module**：利用 LLM 进行语义分析，识别与用户标准匹配的关键词和短语，识别并检索相关招标文件。
    *   **Template Filling Module**：一旦检索到相关文档，使用 LLM 应用 targeted modifications，专注于调整提取的信息以严格遵守指定要求。
    *   **Modifications with Procurement Knowledge Base**：从采购知识库 Incorporate additional knowledge，确保修改后的内容与行业标准和最佳实践保持一致。
*   **评估**：随机选择十个招标文件进行评估，与手动创建的招标文件基准集进行比较，隔离框架的不同组件（模板检索与模板填充）测量其对性能指标的影响。
*   **结果**：框架在段落和表格得分方面均优于传统方法，证明其生成语义正确且结构合规招标文件的能力。
[来源：https://arxiv.org/html/2410.09077v1]

### 2. 提示词工程与框架设计

#### 2.1 提示词设计方法
1.  **角色扮演法**：让 AI 扮演特定角色，显著提升回答的专业性和针对性。例如：“假设你是一位经验丰富的心理咨询师..."。
2.  **结构化模板法**：通过提供固定结构，引导 AI 生成格式规范、内容全面的回答。例如：“请按照以下结构介绍 Python 语言：1、基本概念与特点..."。
3.  **分步思维法**：将复杂任务拆解为多个步骤，引导 AI 逐步思考和分析。例如：“请逐步分析是否应该辞职创业：第一步...第二步..."。
4.  **示例引导法**：通过提供示例，让 AI 理解期望的输出格式和风格。
5.  **反向提问法**：通过让 AI 向你提问来澄清需求，从而获得更精准的结果。例如：“在为我制定健身计划前，请先向我提出 5 个关键问题..."。
6.  **条件约束法**：通过设定限制条件，使输出更加精准和符合要求。例如：“用不超过 300 字，通俗易懂地解释区块链技术..."。
[来源：https://adg.csdn.net/6973116c437a6b40336b7ea4.html]

#### 2.2 主流提示词设计框架
1.  **CRISPE 框架**：
    *   CR（Capacity and Role - 能力和角色）：给 AI 设定具体身份。
    *   I（Insight - 洞察和背景）：提供任务相关的背景信息。
    *   S（Statement - 声明与任务）：直接、清晰地告知 AI 需要完成的具体任务。
    *   P（Personality - 个性与风格）：设定 AI 输出内容的语气、风格和表达特点。
    *   E（Experiment - 实验与示例）：要求 AI 提供多个不同角度或版本的输出。
2.  **CARE 框架**：
    *   C（Context - 上下文）：提供广泛的背景信息，设定场景。
    *   A（Action - 行动）：说明你希望 AI 执行的具体操作。
    *   R（Result - 结果）：明确描述你期望得到的结果或输出。
    *   E（Example - 示例）：提供一个输入和输出的例子。
3.  **TAG 框架**：
    *   T（Task - 任务）：要完成的具体工作。
    *   A（Action - 行动）：需要采取的具体动作。
    *   G（Goal - 目标）：最终希望达成的目的或效果。
4.  **思维链（Chain-of-Thought, CoT）**：要求 AI 将其推理过程一步步地展示出来，适合解决复杂逻辑、数学问题或需要深度分析的任务。
[来源：https://adg.csdn.net/6973116c437a6b40336b7ea4.html]

**技术评分要求 SystemPrompt：**

【权重/分值】：<具体分值或占比，如"30 分”或"40%">
【评分标准】：<详细规则，如"≥95% 得满分，每低 1% 扣 0.5 分”>

### 3. AI Agent 架构与平台实现

#### 3.1 基于腾讯云智能体开发平台 LKE 的实现
*   **模型配置**：支持精调知识大模型、混元大模型、DeepSeek 等。DeepSeek-R1 输入 32k，DeepSeek-V3 为 6710 亿参数的 MoE 模型。
*   **参数设置**：
    *   **温度**：调高温度会使得模型的输出更多样性和随机性（创造性场景 0.8-1.0），降低温度会使得输出的内容更遵循指令（确定性场景 0.2-0.5）。
    *   **Top P**：累计概率，限制模型只选择高概率词汇。设得更大（如 0.95）纳入更多概率相对没那么高的词，生成内容的多样性也就更强。
*   **知识库设置**：
    *   **文档召回数量**：指系统在处理用户问题等请求时，从知识库中检索并返回的相关文档的数量。
    *   **文档检索匹配度**：用于衡量用户输入与知识库中文档之间的匹配程度的阈值。像设置为 0.2，当某个文档与用户输入的匹配度达到或超过 0.2 时，才会被作为相关文档召回。
    *   **检索范围**：支持“且”与“或”逻辑，传入多个参数映射多个标签时，会检索同时包含多个标签的知识或包含任意标签的知识。
*   **工作流**：用户上传文件 (PDF) → 统一归为文本数据类型 → 关键文本信息传入大模型 → 上传相关未整理的文件输入 → 数据格式清洗 → 检索相关政策 → 传入大模型。
*   **输出配置**：流式适合实时性高的场景，非流式适合需要高连贯性和格式化的任务。
[来源：https://cloud.tencent.com/developer/article/2498790]

#### 3.2 零代码搭建方案（Coze+TextIn）
*   **架构**：Coze 负责对话、工作流编排、输出格式、交互体验；TextIn xParse 负责把招标文件解析成结构化可读内容；LLM 负责条款抽取、风险识别、建议生成。
*   **链路**：用户上传招标文件 → Coze 调用 TextIn xParse 插件 → 得到 markdown/结构化结果 → Coze 用大模型做解析结果抽取 → 输出最终“招标解析报告”。
*   **Bot 人设**：输出要结构化，尽量引用原文/页码，风险提示要分级（高/中/低），遇到信息缺失要主动提问澄清。
[来源：https://cloud.tencent.com/developer/article/2631225]

### 4. 投标写作工作流与内容生成

#### 4.1 投标技术方案写作流程（WorkBuddy）
1.  **人工拆招标文件**：先把项目名称、项目背景、服务范围、技术要求、评分标准、交付成果、人员要求、时间要求、售后要求、废标条款摘出来。
2.  **做技术标目录**：1. 项目理解 2. 技术路线 3. 实施方案 4. 项目组织管理 5. 进度计划 6. 质量保障措施 7. 安全与风险控制 8. 培训方案 9. 售后服务 10. 类似项目经验。然后根据评分标准调整顺序和重点。
3.  **逐章生成**：不要一次性让它写完整技术标。更推荐“一章一章写”。每次都带上招标要求、本章目标、我方真实材料、写作限制和输出结构。
4.  **人工核对评分点**：这个评分点有没有回应？回应在哪一章？是否有证明材料？是否出现了不该承诺的内容？是否和商务部分、报价部分冲突？
5.  **统一语言和格式**：请帮我统一以下投标技术方案的语言风格。要求：1. 保持正式、专业、稳健；2. 不改变原意；3. 不新增承诺；4. 删除重复、空泛、口语化表达；5. 保留章节结构。
[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]

#### 4.2 适合 AI 辅助的 5 类章节
1.  **项目理解**：让 AI 帮你把“项目背景、建设目标、需求理解、重点难点、响应策略”串起来。输出结构包括：项目背景、建设目标、需求理解、重点难点、总体响应思路。
2.  **技术路线**：写清楚五件事：为什么这么做、分几个阶段做、每个阶段交付什么、如何保证效果、如何与招标要求对应。按阶段展开，每个阶段说明主要工作、输出成果、保障措施。
3.  **实施计划**：展开成更完整的结构：项目启动、需求调研、方案深化、组织实施、联调测试、验收交付、后续服务。按阶段说明工作内容，每个阶段包含目标、关键任务、交付成果。
4.  **质量保障措施**：拆成组织保障、制度保障、过程控制、成果审核、问题整改、风险预防几个层面。每一项措施都要写出具体动作。
5.  **售后服务与响应保障**：结构包括：服务体系、响应机制、问题处理流程、服务记录与改进。只基于提供的信息写，不新增服务承诺。

### 5. 评标辅助与风险控制

#### 5.1 AI 辅助评标 10 个步骤
1.  **明确 AI 辅助评标需求与辅助边界，设定审查清单**：列出资质审查、技术响应、商务报价三大板块的具体审查项。将资质年检过期、社保缺失等硬性否决项设为 AI 优先筛查目标。
2.  **招标文件核心要素拆解，提取 AI 识别基准**：提取资质要求、技术参数带"★"号条款、商务计分规则。转化为结构化清单，剔除模糊修饰语，转化为明确参数阈值。
3.  **构建 AI 辅助评标高效技巧之提示词框架**：按照“角色设定 + 任务目标 + 输入数据 + 输出格式 + 约束条件”顺序编写提示词。约束条件必须注明“仅依据提供的文本客观比对，禁止推测与主观评价”。
4.  **投标文件数据提取与结构化，统一输入格式**：将各投标人文件导入格式转换工具，确保文本与表格可选中提取。对大体积文件按资质、技术、商务拆分上传。
5.  **资质符合性智能初审，执行硬性条件筛查**：指令 AI 逐项核对营业执照、资质证书、财务报表等是否满足要求。提取证书编号与有效期，排查过期资质。
6.  **技术方案偏离度分析，定位星号条款响应**：指令 AI 逐条比对技术响应，重点筛查正偏离、完全响应与负偏离项。针对带"★"号实质性要求，指令 AI 提取原文描述并判定是否满足。
7.  **商务报价合理性评估，校验算术性错误**：指令 AI 核对大小写金额是否一致，分项报价汇总是否等于总报价，识别是否存在算术性错误。按照招标文件载明的价格分计算公式，指令 AI 计算该投标人的价格得分。
8.  **评标质疑点与风险提示生成，汇总异常指标**：指令 AI 汇总前述步骤中的异常项与负偏离项，生成独立的风险提示清单。要求 AI 识别低于成本价的异常报价、技术方案照搬照抄的雷同表述、资质文件存疑的盖章页等特征。
9.  **人工复核与 AI 输出结果交叉验证，修正误判项**：评标专家逐项审阅 AI 生成的审查表格与风险清单。调阅原始投标文件，对 AI 标记的“不合格”或“负偏离”项进行重点人工比对。
10. **评标报告辅助生成与归档，固化评审轨迹**：指令 AI 按照标准评标报告模板，生成包含初步审查情况、详细审查情况、澄清情况与推荐中标候选人的报告草稿。必须在报告中注明"AI 仅作辅助工具，评审结论由评标委员会负责”的声明。
[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]

#### 5.2 投标废标风险筛查与合规校验
*   **风险类型**：资格不匹配/隐性门槛、付款条件不利/履约风险、交付周期不合理、“一票否决”/废标点、合同条款冲突/模糊表述。
*   **筛查重点**：暗标格式、签章、超链接等细节极易遗漏，随时可能面临废标风险。
*   **合规建议**：AI 通过知识库中的《招投标法实施条例》，自动识别“围标”“串标”相关表述，并在生成内容中标注合规建议。通过预设评分标准模板，AI 在生成“投标报价”章节时自动校验价格合理性，标记可能触发“低价风险警示”的条目。
[来源：https://cloud.tencent.com/developer/article/2631225] [来源：https://www.53ai.com/news/zhinenghuagaizao/2025022571652.html]

#### 5.3 AI 辅助评标避坑方法
*   **坑点 1**：盲目信任 AI 审核结论，导致不合规投标人漏网。解决方案：将 AI 设定为辅助筛查工具，所有符合性审查与强制性标准判定必须由评标专家签字确认。
*   **坑点 2**：投标文件格式混乱导致提取失败。解决方案：操作前先确认文档文本可选中，若为纯图片扫描件，需经过通用 OCR 工具识别转换为文本后输入。
*   **坑点 3**：提示词缺乏评标标准约束，AI 按通用认知评价。解决方案：在提示词中完整嵌入评标办法的权重、算分公式与废标条款。
*   **坑点 4**：商业敏感数据外泄风险。解决方案：优先选择支持私有化部署或提供企业级数据隔离的通用 AI 工具，关闭模型训练数据复用选项。
*   **坑点 5**：忽略行业特殊评标规则，AI 判定偏离实际。解决方案：在步骤 2 拆解要素时，将特定行业的资质互认规则、特殊加分项作为独立约束条件显性写入提示词。
*   **坑点 6**：人工复核流于形式，仅做抽样检查。解决方案：制定强制复核清单，要求对 AI 标记的负偏离项、质疑问询项及排名前三位候选人的核心资料必须进行 100% 人工原始文件比对。
[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]

## 推荐工作流

### 1. 组合使用工具建议
*   **解析层**：使用 TextIn xParse 或类似 OCR/文档解析工具处理 PDF/扫描件，输出 Markdown+ 结构化信息 + 页码定位。[来源：https://cloud.tencent.com/developer/article/2631225]
*   **模型层**：选用支持长上下文（如 32k+）且中文能力强的模型（如 DeepSeek-R1/V3，混元长文本版），温度设置为 0.2 以下以适应政务/审核场景。[来源：https://cloud.tencent.com/developer/article/2498790]
*   **应用层**：使用 Coze 或腾讯云 LKE 搭建工作流，编排“解析→抽取→生成→审查”链路。[来源：https://cloud.tencent.com/developer/article/2631225]
*   **知识库**：建立企业私有知识库（历史标书、资质、人员、政策），并在生成时绑定特定章节。[来源：https://www.53ai.com/news/zhinenghuagaizao/2025022571652.html]

### 2. 具体执行步骤
1.  **准备阶段**：人工拆解招标文件，提取资质要求、技术参数带"★"号条款、商务计分规则，转化为结构化清单。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]
2.  **解析阶段**：上传招标文件至解析工具，获取结构化文本及页码索引。[来源：https://cloud.tencent.com/developer/article/2631225]
3.  **配置阶段**：在 AI 平台配置知识库，将投标案例、人员信息等文件勾选“允许 AI 引用”，并绑定到标书目录章节。[来源：https://www.53ai.com/news/zhinenghuagaizao/2025022571652.html]
4.  **生成阶段**：按章节逐章生成（项目理解→技术路线→实施计划→质量保障→售后服务），使用结构化提示词模板，禁止一次性生成全文。[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]
5.  **审查阶段**：AI 自动检测格式规范、评分匹配度、废标条款，输出风险清单（高/中/低）及原文引用。[来源：https://www.kuaibiaobao.com]
6.  **复核阶段**：人工对 AI 标记的负偏离项、质疑问询项及排名前三位候选人的核心资料进行 100% 人工原始文件比对。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]

### 3. 配置方法
*   **温度值**：创造性任务提高温度值（0.8-1.0），事实性任务降低温度值（0.2-0.5）。[来源：https://adg.csdn.net/6973116c437a6b40336b7ea4.html]
*   **Top P**：涉及容错率相当低的审核效验场景，Top P 建议设小，限制模型选词范围。[来源：https://cloud.tencent.com/developer/article/2498790]
*   **上下文记忆**：设置输入给大模型作为 prompt 的上下文对话历史轮数，轮数越多，多轮对话的相关性越高，但消耗的 token 也越多。[来源：https://cloud.tencent.com/developer/article/2498790]

## 适用场景

1.  **大规模招标项目**：项目规模扩大，文书复杂度和工作量增加，传统人工写作方法难以应对效率低下、易出错、文档一致性差的问题。[来源：https://cloud.tencent.com/developer/article/2498790]
2.  **标准化内容生成**：许多招标文书包含大量标准化的内容，如项目介绍、公司资质、投标要求等，AI 可根据项目基本信息自动生成这些标准化部分的初步草稿。[来源：https://cloud.tencent.com/developer/article/2498790]
3.  **多轮修改与反馈场景**：招标文书的写作过程中，经常需要多次修改和反馈，AI 可以根据用户的反馈，快速生成修改后的文书内容，并进行内容优化。[来源：https://cloud.tencent.com/developer/article/2498790]
4.  **企业私有知识库场景**：需要将企业历史标书、资质人员信息、政策文件等结构化知识嵌入生成流程，解决传统标书内容泛化、专业性不足痛点。[来源：https://www.53ai.com/news/zhinenghuagaizao/2025022571652.html]
5.  **评标辅助审查**：企业采购与招投标管理场景，帮助从业者掌握 AI 辅助评标怎么操作的核心流程，大幅提升评标效率与准确性，规避合规风险。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]

## 不适用场景

1.  **法定评标决策**：AI 仅作为信息提取与比对预处理工具，不具备法定评标决策权，所有结论需人工复核，最终废标决定权归属评标委员会。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]
2.  **前段策略判断**：前段涉及投标策略和合规判断（读招标文件、拆评分点、判断响应策略），要由人把关，AI 不适合直接判断废标条款，不适合替你确认资质是否满足。[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]
3.  **真实资质承诺**：类似项目经验、人员证书、资质能力、服务网点，都必须真实存在。AI 可以帮你改写，不能帮你编造。不能让 AI 承诺做不到的指标（如"7×24 小时驻场”"1 小时到场”）。[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]
4.  **纯图片扫描件无 OCR 处理**：若为纯图片扫描件，需经过通用 OCR 工具识别转换为文本后输入，避免 AI 解析空白或乱码，直接丢给大模型常会翻车。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html] [来源：https://cloud.tencent.com/developer/article/2631225]

## 风险与约束

### 1. 衔接点风险
*   **描述**：文档过长会被截断，信息丢失；无法可靠解析扫描件、复杂表格和特殊版式。
*   **应对措施**：先把文档变成结构化可理解的内容，再让大模型做抽取和风控。使用 TextIn xParse 等工具把复杂结构转换成更适合大模型处理的 Markdown+ 结构化信息。[来源：https://cloud.tencent.com/developer/article/2631225]

### 2. 上下文风险
*   **描述**：多段落协同生成与逻辑一致性风险。如果在“技术方案”里承诺要采用某种技术架构，那么在“实施计划”部分便要有与之相匹配的时间安排、资源调度说明。
*   **应对措施**：DeepSeek 在调用时，可将已生成的文本或项目概要以上下文形式输入，让模型在输出后续章节时自动保证逻辑匹配。发现某些段落与前文描述不符，将这些关键信息再次输入模型进行“二次校验生成”。[来源：https://cloud.tencent.com/developer/article/2498790]

### 3. 测试与复核风险
*   **描述**：人工复核流于形式，仅做抽样检查。AI 生成的内容可能很顺，顺不代表准确。
*   **应对措施**：制定强制复核清单，要求对 AI 标记的负偏离项、质疑问询项及排名前三位候选人的核心资料必须进行 100% 人工原始文件比对。最终判断必须人工完成。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html] [来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]

### 4. 规格漂移风险
*   **描述**：提示词缺乏评标标准约束，AI 按通用认知评价。忽略行业特殊评标规则，AI 判定偏离实际。
*   **应对措施**：在提示词中完整嵌入评标办法的权重、算分公式与废标条款，强制限制 AI 的自由裁量权与幻觉。将特定行业的资质互认规则、特殊加分项作为独立约束条件显性写入提示词。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html]

### 5. 数据安全风险
*   **描述**：招投标数据涉及商业机密，需明确 AI 工具的数据处理机制，严禁将敏感数据暴露于无保障环境。
*   **应对措施**：优先选择支持私有化部署或提供企业级数据隔离的通用 AI 工具，关闭模型训练数据复用选项，标后及时清除云端会话记录。[来源：https://www.163.com/dy/article/KVAMCJMO0556M2QN.html] [来源：https://www.kuaibiaobao.com]

### 6. 过度承诺风险
*   **描述**：让 AI 承诺做不到的指标，只要写进投标文件，就可能成为合同履约压力。
*   **应对措施**：在提示词中约束“不得新增我没有提供的资质、案例、人员、设备和服务承诺”，“如果信息不足，请用保守表达”。[来源：https://www.53ai.com/news/neirongchuangzuo/2026060363284.html]
