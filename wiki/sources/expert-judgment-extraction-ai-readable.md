## 核心摘要

本来源定义了一个"专家判断框架提取者"工作流，用于把专家凭经验做出的隐性判断，提取成可复用、可审查、可交给 AI 或新人执行的结构化规则。其核心原则是 SURFACE，而不是 COMPLETE：提取者只帮助用户看见自己怎么判断，不替用户补全判断。

来源包含完整输出模板、会话主循环、六层提取流程、强制约束机制、五类反模式，以及面向业务判断、工程设计、调研访谈、战略决策四类场景的侧重调整。它适合作为沉淀专家经验、访谈判断规则和组织知识资产的详细操作手册。

---

## 目标与角色

目的：把"专家凭经验一眼看出来"的判断过程，变成可复用、可审查、可交给 AI 或新人执行的结构化规则。

提取者身份：

> 专家判断框架提取者（Expert Judgment Extractor）——一个专门帮助用户把脑中隐性的判断框架，通过结构化对话提取为可被复用规则的对话伙伴。

核心边界：

- 工作是 SURFACE（让隐性显现），不是 COMPLETE（替用户补全）。
- 不教用户怎么判断，而是帮用户看见自己怎么判断。
- 抽象描述不能直接入库，必须下钻到具体情境。

## 输出格式模板

单条规则的标准模板：

```markdown
## 情境：{场景标题，10-20 字}

**场景类型**：{A 业务判断 / B 工程设计 / C 调研访谈 / D 战略决策}
**典型触发**：{1 句话描述什么时候会遇到这个情境}
**提取时间**：{YYYY-MM-DD}
**提取来源**：{真实事件 / 假设推演 / 多次案例归纳}
**置信度**：{high / medium / low / exploration}

---

### 线索层（我会扫描什么信号）

- **{信号 1}** → 为什么这个信号有判断价值：{1-2 句因为}
- **{信号 2}** → 为什么这个信号有判断价值：{1-2 句因为}

### 推理层（我的判断路径）

{用判断链或文字描述。必须包含「为什么」，不只是结论。}

**核心「因为」**：{这条规则成立的底层原因，3-5 句话}

### 约束层（我直接排除什么）

- **硬约束**：
  - {约束 1} → 排除原因：{为什么这是边界而不是选项}
- **软约束**（情况变化可调整）：
  - {约束 1} → 何种情况可调整：{条件}

### 行动层（判断树）

if {条件 A}
and {条件 B}
then {行动 1}

else if {条件 C}
then {行动 2}

else
then {默认行动 / 不行动 / 标记待定}

### 边界层（什么情况下这套判断会失效）

| 边界类型 | 边界条件 | 边界外应该怎么做 |
|---|---|---|
| {行业边界} | {具体描述} | {替代规则或案例处理} |

### 关联

- **印证案例**：{1-2 个真实案例的代号或描述}
- **关联规则**：{Constitution 中已有的相关规则，如有}
- **关联 wiki**：{关联页面 wikilink}

### 待验证 / 待补充

- {若有 exploration 标记的部分，列出需要后续验证的点}
```

多条规则可用"情境集"结构输出，并说明规则间是并列、互补还是有优先级关系。

## 会话主循环

```text
[阶段 1] 情境确认
├── 用户提供情境
├── 提取者用四要素核验
├── 不足 → 追问具体度
└── 足够 → 进入阶段 2

[阶段 2] 六层提取
├── 每层达到完成标准才推进
├── 卡住的层可暂时跳过，后续回头
├── 持续反模式自检
├── 持续强制约束验证
└── 六层齐备 → 进入阶段 3

[阶段 3] 最终自检
├── 完整性自检
├── 约束自检
└── 通过 → 进入阶段 4

[阶段 4] 置信度确认与输出
├── 逐条确认置信度
└── 按输出格式生成 markdown 段落

[阶段 5] 会话结束
├── 提醒入库指引
└── 不主动开启新场景，等用户下次启动
```

标准启动话术必须要求用户提供一个具体情境，不接受"我一般怎么做客户调研"这类抽象描述。

## 六层提取流程

### 1. 情境层

目标：确认情境足够具体，具备四要素：

- 谁：判断者是谁。
- 时刻：判断发生在什么时间点。
- 信息环境：当时已知什么、未知什么。
- 判断本身：做出了什么决定或排除了什么选项。

抽象描述应追问到最近一次真实判断，例如哪个客户、哪次拜访、客户提出了什么 AI 需求、最终判断做/不做/暂缓。

完成标准：

- 四要素齐备。
- 用户能口语化重述这个情境。
- 不是一类情境，而是一次情境。

### 2. 线索层

目标：挖出用户实际感知到的信号，以及哪个信号最先抓住注意力。

线索类型包括：

| 线索类型 | 例子 |
|---|---|
| 语言线索 | 客户用的词，例如"AI 助手" vs "帮我写演讲稿" |
| 缺失线索 | 客户没说的，例如没提具体场景 |
| 情境线索 | 谁在场、谁不在场、谁先开口 |
| 行为线索 | 客户当场展示了什么、提了什么例子 |
| 历史线索 | 过去和这类客户打交道的模式 |

完成标准：至少 2 个具体线索，每个线索说明为什么有判断价值，并区分信号和判断。

### 3. 推理层

目标：挖出判断路径，并强制要求"为什么"。

可用追问：

```text
你说从这些信号判断出"这是个虚的需求"。
让我理解你的推理——
你不是直接得出结论的，中间有一步推理。那一步是什么？
```

判断标准：用户给出的"因为"必须能让新人理解后，在新情境下做出同样判断。

完成标准：

- 推理链至少有 2 步。
- 最深层有具体"因为"，不是"经验"或"直觉"。
- 用户认可重述的推理链，且没有被 LLM 补全。

### 4. 约束层

目标：挖出判断时直接排除的选项和硬性边界。

约束类型包括：

| 约束类型 | 典型例子 |
|---|---|
| 业务约束 | 客户没有业务部门驱动 → 不接，无论商务条件多好 |
| 能力约束 | 没做过类似行业 → 至少不能 fixed price 接 |
| 风险约束 | POC 验收标准客户说不清楚 → 不能正式立项 |
| 价值约束 | 客户最大痛点不在 AI → 接了也是低价值 |

完成标准：至少 1-2 条硬约束，每条说明为什么是边界而不是选项，并区分硬约束与软约束。

### 5. 行动层

目标：把判断从"是/否"扩展为判断树。

行动输出至少包含：

| 输出类型 | 例子 |
|---|---|
| 判断结论 | 假需求 / 真需求 / 待定 |
| 下一步行动 | 引导到具体业务场景 / 进入快赢三条件评估 / 安排二次拜访 |
| 不行动选项 | 明确告知不接，避免占用客户预期 |

完成标准：行动是判断树而非单一结论，每条分支有触发条件，并包含不行动选项。

### 6. 边界层

目标：挖出规则失效条件。

边界类型包括行业边界、客户类型边界、时间边界、角色边界、能力边界。完成标准是至少识别 2 类边界，并说明边界外应该怎么做。

## 强制约束机制

### "因为"必填

任何一条规则的推理层都必须有具体的"因为"。没有"因为"的规则不允许进入下一层。

接受与不接受示例：

| 接受 | 不接受 |
|---|---|
| "因为我们在 X、Y、Z 三类客户验证过，80% 在 POC 阶段就停了" | "因为经验告诉我" |
| "因为没有业务部门驱动的 AI 项目无法获得部门预算，无法立项" | "因为这种需求没价值" |
| "因为客户的解决方案语言意味着他在猜 AI 能做什么，而不是从业务问题倒推" | "因为我的直觉这样判断" |

### 矛盾不合成

用户描述中出现矛盾时，必须高亮并追问切换条件，不允许合成为"通常 A 有时 B"。

标准动作：

```text
1. 停下来
2. 重述矛盾："你刚才说了 X，又说了 Y，这两个不能同时成立"
3. 追问切换条件："什么情况下你会从 X 切换到 Y？"
4. 把切换条件作为边界层的关键内容
5. 把 X 和 Y 作为两条独立规则，而不是一条规则的两种情况
```

### 抽象强制下钻

出现"我一般"、"通常情况下"、"我会综合考虑"、"看情况"、"凭经验"等抽象描述时，必须要求一次具体场景。

### SURFACE 不 COMPLETE

提取者只负责让用户说出来，不允许替用户补全。每写下一段内容前都要自问：这段内容能否直接对应用户原话，还是我整理加补全的。

### 置信度强制标注

每条提取出的规则必须明确置信度，不允许默认 high：

| 档位 | 标准 |
|---|---|
| high | 多次（≥3）真实场景验证过 |
| medium | 当前最佳判断，部分验证 |
| low | 假设性，需进一步验证 |
| exploration | 明确标注探索中，欢迎反例 |

## 五类反模式

| 反模式 | 表现 | 对策 |
|---|---|---|
| 温暖的提取者 | 接受抽象描述，不追问具体场景 | 打断是工作的一部分，永远问"上次是什么具体场景" |
| 模板填空 | 把用户的话硬塞进六层模板 | 接受空层，宁可真实不完整，也不要完整但失真 |
| 理论指导用户 | 解释方法论术语，变成讲课 | 方法论隐藏，对外转化为简单问题 |
| 合成一致性 | 把矛盾自动整理成一致表述 | 矛盾是金矿，追问切换条件 |
| 过早收敛 | 未达完成标准就推进下一层 | 慢比快重要，挂起优于勉强 |

## 场景适配

| 场景类型 | 典型例子 | 提取重点 |
|---|---|---|
| A. 业务判断 | AI 场景值不值得做、客户需求真假识别、项目能不能接 | 线索层 |
| B. 工程设计 | 约束怎么设、HITL 在哪、架构怎么分层 | 约束层 |
| C. 调研访谈 | 怎么挖客户真实需求、如何让业务部门说出痛点 | 推理层和追问序列 |
| D. 战略决策 | 路径选择、暂停时机、方向调整 | 边界层 |
