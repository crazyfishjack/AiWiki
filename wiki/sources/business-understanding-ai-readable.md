## 核心摘要

本来源定义了"读懂业务"的结构化框架：业务等于客户靠什么赚钱、怎么赚钱、遇到什么阻碍三者的组合。它进一步把业务理解拆成主体、价值、流程、组织、技术、环境六个调研维度，并要求不同置信度的结论必须对应不同表达方式。

来源还给出业务理解的三档成熟度：初级只能说出主营业务，中级能描述核心流程与关键节点，高级能说清痛点背后的组织利益和决策逻辑。该框架适合 AI 或新人执行客户调研时作为事实收集、置信度标注和业务洞察分层的清单。

---

## 业务理解公式

```text
业务 =
客户靠什么赚钱（价值层，包括主体和价值） ×
怎么赚的（流程层，包括流程、组织和技术） ×
遇到什么阻碍（压力层，也就是环境）
```

---

## 置信度表达规则

调研必须严格标明数据来源，优先使用权威数据和链接。按证据强弱使用不同表述：

- 高置信："根据[来源]，……"
- 中置信："现有资料显示……但建议确认"
- 低置信："目前没有充分资料支撑，建议核实"
- 无支撑：直接拒答，或回答"基于现有资料无法回答"

原则：需要在有实际数据支撑的情况下才能进入更高阶段说明。严禁臆想、胡编乱造；没有实际数据时，不做更高阶段判断。

## 六个调研维度

### 主体

| 要素 | 具体内容 |
|---|---|
| 企业性质 | 央企/国企/民企/外资，决定决策逻辑 |
| 股权结构 | 谁是实际控制人、上级主管单位 |
| 主营牌照 | 持有什么经营资质、受哪个部委监管 |
| 规模体量 | 营收/资产/员工人数/子公司数量 |

### 价值

| 要素 | 具体内容 |
|---|---|
| 收入来源 | 主营业务收入结构（各板块占比） |
| 盈利模式 | 卖产品/卖服务/卖资源/卖数据 |
| 核心产品 | 旗舰产品是什么，利润从哪来 |
| 客户群体 | B2B / B2G / B2C |

### 流程

| 要素 | 具体内容 |
|---|---|
| 核心业务流程 | 从原料/线索到最终交付的全链条 |
| 关键作业节点 | 哪些节点是卡脖子/高风险/高成本 |
| 信息流转方式 | 数据在哪里产生、流向哪里、如何使用 |
| 决策机制 | 什么事谁来拍板、依据什么信息 |
| 知识载体 | 经验存在人脑还是文档还是系统里 |

### 组织

| 要素 | 具体内容 |
|---|---|
| 部门分工 | 各部门职责边界，谁管生产谁管 IT |
| 人员结构 | 一线操作工/工程师/管理层比例 |
| 能力短板 | 哪类人才稀缺，哪些依赖外包 |
| 利益格局 | 谁是项目的推动者、谁是阻力 |

### 技术

| 要素 | 具体内容 |
|---|---|
| 已有系统 | ERP / MES / OA / PLM / SCADA 等 |
| 系统成熟度 | 老系统还是近年上线的 |
| 数据现状 | 数据在哪里、有没有治理、能不能用 |
| 技术偏好 | 是否有信创要求、倾向哪个云 |
| 集成难度 | 各系统是否打通、接口是否标准 |

### 环境

| 要素 | 具体内容 |
|---|---|
| 行业政策 | 主管部委的监管要求、十五五政策 |
| 市场竞争 | 同行在做什么，客户的竞争焦虑 |
| 成本压力 | 人力成本上升/原材料波动 |
| 安全合规 | 事故/罚款/审计压力 |
| 上级要求 | 集团下达的数字化 KPI |

## 业务理解段位

| 段位 | 表现 | 结果 |
|---|---|---|
| **初级** | 能说出客户主营业务是什么 | "他们是做电力设备的" → 只能聊皮毛 |
| **中级** | 能描述核心业务流程的关键节点 | "他们变压器质检现在全靠人工，每班 3 人" → 能找到痛点 |
| **高级** | 能说清楚痛点背后的组织利益和决策逻辑 | "质检部门不愿上 AI 怕裁员，但生产副总有降本 KPI，这是突破口" → 能推动成交 |
