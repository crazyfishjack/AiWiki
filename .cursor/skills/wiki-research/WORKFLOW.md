# Wiki 调研工作流

## 1. 定位 Wiki 上下文

1. 按以下顺序加载配置：
   - 传给 `scripts/four_tool_research.py` 的 `--config`
   - `WIKI_RESEARCH_CONFIG`
   - `SKILL.md` 同目录的 `config.local.json`
2. 读取 `wiki_root`。
3. 读取 `wiki_root` 下的 `SCHEMA.md`。
4. 读取 `wiki/_index.md`。

如果任一必需文件缺失，立即停止，并请用户修正配置或 Wiki 路径。

## 2. Wiki 优先查询

遵循 `SCHEMA.md` 的 Query 工作流：

1. 将用户问题解析为核心实体、概念和预期页面类型。
2. 优先检索 `wiki/_index.md`。
3. 如果存在候选页，先读取直接命中的页面。
4. 只在必要时读取关联页，优先读取 `contradicts`、`supersedes` 和直接主题关联。
5. 回答时使用 `[[wikilinks]]`，引用来源支撑的知识时标注 `[[sources/...]]`。

如果 Wiki 已足够回答，到这里停止。不要调用外部调研脚本，也不要创建 raw 文件。

## 3. 外部调研触发条件

仅在满足以下任一条件时使用外部调研：

- `wiki/_index.md` 没有相关条目。
- 现有页面过时、低置信、有矛盾或内容明显不足。
- 用户明确要求进行新的外部调研。
- 用户要求生成新的 raw 调研文档。

运行：

```bash
python .cursor/skills/wiki-research/scripts/four_tool_research.py --query "调研主题"
```

推荐参数：

```bash
python .cursor/skills/wiki-research/scripts/four_tool_research.py \
  --query "调研主题" \
  --mode standard \
  --raw-dir raw/articles
```

## 4. 四工具路由

脚本遵循项目四工具 SOP：

- Tavily：广度网页、新闻、最新动态、中文覆盖。
- Exa：语义搜索、学术论文、技术内容、公司和人物信息。
- Firecrawl：普通页面精读、搜索 + 全文、结构化萃取、整站爬取。
- Jina：仅用于学术 PDF / 结构化学术页面和候选重排序。

质量优先默认值：

- Tavily 使用 advanced search，并返回 raw markdown content。
- Exa 使用 deep reasoning，并返回 text、highlights 和 summary；highlights 默认 4000 字符，仅用于候选发现和筛选。
- Firecrawl 使用 markdown 和 `onlyMainContent`。
- Jina readerlm-v2 仅用于学术域名。
- 标准模式默认精读最多 10 个 URL。
- 普通网页由 Firecrawl 精读，Firecrawl/Jina 精读正文默认完整写入证据包，不按字符数截断。

## 5. 证据包与最终 raw 调研报告

脚本会先在 `raw/` 下写入一篇 Markdown 证据包草稿。它只是外部检索和精读结果的载体，不是最终可用调研报告。

默认目录路由：

- 通用调研：`raw/articles/`
- 学术调研：`raw/papers/`
- 项目相关调研：`raw/projects/`
- 用户指定 `--raw-dir` 时，以用户指定目录为准。

脚本生成的证据包应包含：

- 调研标题和问题
- 生成日期
- 工具模式
- 候选信源 URL 清单
- 完整精读结果
- 工具运行摘要
- AI 综合提示

Agent 必须继续执行“AI 编译”步骤，基于证据包中的完整精读结果编写为一个新的最终中文 raw 调研报告，并删除对应证据包。若高可信来源精读失败、正文为空、明显只包含导航/付费墙/反爬内容，或工具返回结果不足以覆盖正文，Agent 必须二次精读或标注不可验证。

最终报告必须包含：

- `## 核心结论`：3-7 条可复用结论，每条带来源 URL 和可信度。
- `## 内容整理`：保留所有有用的信息，系统化，结构清晰，逻辑正确。
- `## 推荐工作流`：按步骤说明怎么结合工具使用，明确 Cursor 执行点。
- `## 适用场景`：什么情况下值得使用 OpenSpec + Superpowers + Cursor。
- `## 不适用场景`：什么时候会过度工程或拖慢开发。
- `## 风险与约束`：衔接点、上下文丢失、测试跳过、规格与代码漂移等风险。
- `## 信源质量门控记录`：列出门控规则、保留信源、剔除信源、来源等级、保留/剔除原因和后续处理。
- `## 来源与可信度`：列出关键来源、来源类型、可信度和支撑内容。
- `## 对于可信度较高的来源逐来源总结`：对高可信来源逐篇整理核心内容、可用事实、局限和适合入库的知识点。
- `## 跨源矛盾检测结论`：检测核心数字、日期、功能描述、因果判断、市场/公司事实；未发现矛盾也必须明确写出。
- `## 矛盾与待验证问题`：显式保留冲突或证据不足的点。
- `## 原始证据摘录`：保留必要摘录或完整证据片段，供后续 Ingest 复核。

关键事实必须使用 `[来源: URL]` 标注。可信度使用：

- 高：至少一个高质量来源直接支撑，或多个来源一致。
- 中：单一可信来源支撑，或需要轻度推断。
- 低：来源弱、存在矛盾、内容过时或需要人工验证。

没有 URL 支撑的内容只能出现在“待验证问题”中，不得作为正文事实写入。

如果证据包中的重要来源正文为空、抓取失败、出现 `[truncated]`，或明显只包含页面开头、目录、导航而不足以总结正文，Agent 必须对该来源二次精读全文后再生成最终报告，不得只依赖不完整内容。

### 信源质量门控记录模板

最终 raw 调研报告必须按以下结构输出，不得只写宽泛说明：

```markdown
## 信源质量门控记录
### 门控规则
- Tavily score < 0.4：剔除
- 已知低质域名：剔除
- 重复 URL：合并保留最高分结果
- Exa 学术/语义结果：默认保留，但进入来源等级评估
- C/D 级来源不得作为唯一结论依据
- 入库映射：A/B → `source-quality: high`；C → `source-quality: medium`；D → `source-quality: low`

### 保留信源
1. [标题](URL)
   - 工具：Tavily / Exa
   - 分数：0.xx / Exa 默认保留
   - 来源等级：A / B / C / D
   - 入库映射：`source-quality: high / medium / low`
   - 保留原因：官方文档 / 一手数据 / 权威媒体 / 学术论文 / 与主题高度相关
   - 后续处理：进入精读 / 进入 rerank / 仅作背景参考

### 剔除信源
1. [标题](URL)
   - 工具：Tavily
   - 分数：0.xx
   - 来源等级：A / B / C / D
   - 剔除原因：score 低于 0.4 / 个人问答站 / 电商页 / SEO 聚合页 / 重复 URL / 内容与主题弱相关
```

### 跨源矛盾检测结论模板

最终 raw 调研报告必须用实际检测结果替换模板占位；不得静默合并冲突信息。

```markdown
## 跨源矛盾检测结论
### 检测范围
- 已精读来源数量：N 个
- 检测对象：核心数字、日期、功能描述、因果判断、市场/公司事实
- 检测时间：YYYY-MM-DD

### 检测结果
结论：未检测到跨源矛盾，可进入综合阶段。

或：

结论：检测到 X 处跨源矛盾，综合输出时必须保留双方表述，不得静默合并。

### 矛盾项 1：[简短标题]
- 矛盾描述：[具体什么矛盾]
- 来源 A：[URL]
  - 原文引用：“……”
  - 来源等级：A / B / C / D
  - 发布时间 / 数据日期：YYYY-MM-DD
- 来源 B：[URL]
  - 原文引用：“……”
  - 来源等级：A / B / C / D
  - 发布时间 / 数据日期：YYYY-MM-DD
- 初步判断：
  - 倾向：[来源 A / 来源 B / 暂不倾向]
  - 理由：[官方一手来源优先 / 数据更新 / 方法更透明 / 与中文本地来源一致]
- 综合输出要求：
  - 必须写成“来源 A 说 X，来源 B 说 Y，建议人工核实”
  - 禁止取平均值、合并成第三种说法、只保留其中一方
```

## 6. 必须询问是否入库

最终中文 raw 调研报告完成并删除对应证据包后，必须询问：

```text
中文调研报告已保存到 raw 文件：[path]。是否按 SCHEMA.md 的 Ingest 流程将它入库？
```

允许的结果：

- 用户确认：对该 raw 文件执行 `SCHEMA.md` Ingest。
- 用户拒绝或表达不明确：报告 raw 文件路径后停止。

在用户确认前，不得更新 `wiki/`、`wiki/_index.md`、`wiki/_graph.md` 或 `wiki/_log.md`。

## 7. 交接给 SCHEMA Ingest

如果用户确认，执行 `SCHEMA.md` 的完整 9 步 Ingest 工作流：

1. 读取 raw 文件。
2. 读取当前 Wiki 状态。
3. 创建 `wiki/sources/` 来源摘要页。
4. 识别并更新 / 新建知识页。
5. 处理关联关系。
6. 更新 `_index.md`。
7. 同步 `_graph.md`。
8. 回填来源摘要页。
9. 追加 `_log.md` 并输出摘要。

遵守 `SCHEMA.md` 的所有质量规则，尤其是：

- 来源必须可追溯。
- raw 调研报告中的 A/B/C/D 来源等级入库时必须映射为 `source-quality`：A/B → high，C → medium，D → low。
- 矛盾必须显式记录。
- `raw/` 创建后是不可变来源层。
- `_log.md` 只能追加。
- 没有明确工作流触发，不得写入。

## 8. 错误处理

- 缺少配置：告诉用户尝试读取了哪个配置路径。
- 缺少 Key：只说明缺失的字段名。
- API 失败：若部分结果有价值则保留，并标注失败工具。
- 没有外部结果：除非已有有意义内容，否则不创建 raw 文件。
- 脚本已生成 raw 但入库失败：保留 raw 文件，并说明失败的 SCHEMA 步骤。
