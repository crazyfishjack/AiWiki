### 1.2 目前技术思路纪要中的想法

以下为项目内部技术思路纪要中的当前设想，不是外部行业事实；该来源为本地目标说明文档，无公开 URL，因此仅作为“项目输入”，不作为行业证据。

当前思路把证书包放在 `M5 查询、AI 检索与证书包` 模块内：用户可通过传统筛选、自然语言检索、粘贴招标要求或微信聊天记录获得推荐资质清单，并把推荐结果一键加入证书包；证书包是一组资质的命名组合，支持保存、复用、共享；加入方式包括手动勾选、按类别筛选、按标签筛选、AI 推荐加入；初始预设包包括华为项目投标包、政府采购标准包、公司能力宣传包；导出范围包括勾选证书、整个证书包、全量资质，格式以 PDF 为主；带水印版允许按查看权限导出，无水印版需审批通过后导出。项目还计划把资质能力通过 RESTful API 提供给投标助手等系统消费。

内部设想已经覆盖“组合、复用、共享、导出、水印、审批、API”的主要方向，但还需要补强：包级版本快照、目录页与排序规则、材料完整性校验、有效期校验、包内材料权限继承、导出任务审计、模板化封面/目录/分隔页、服务端文档生成 API、包级审批与下游消费契约。

## 二、核心结论

1. 证书包 / 材料包不应被设计成“若干文件的下载清单”，更成熟的行业做法是把一组文件作为一个可管理的交付物或工作产品：有包级元数据、默认模板、允许的内容类型、共享字段、工作流、权限和版本快照。Microsoft SharePoint Document Sets 明确定义为“一组可作为单一实体管理的相关文档”，支持默认内容、共享元数据、工作流和包级版本快照。[来源: https://support.microsoft.com/en-au/office/introduction-to-document-sets-3dbcd93e-0bed-46b7-b1ba-b31de2bcd234][来源: https://support.microsoft.com/en-us/office/create-and-manage-document-sets-c71d5796-d559-48de-b1b3-42383bdd13ea]

2. 投标材料库 / RFP 响应系统的最佳实践是“结构化资料库 + 标签/元数据搜索 + 模板化导出 + 协作任务 + 审批与审计”。Flowcase 将 RFP 技术关键能力概括为集中内容库、智能搜索过滤、协作任务、模板与导出、权限审计、系统集成、版本历史与内容锁定；PandaDoc 强调模板、内容库、审批流、审计追踪和 API 自动生成文档。[来源: https://www.flowcase.com/blog/12-features-to-look-out-for-when-evaluating-rfp-technology][来源: https://www.pandadoc.com/rfp-software/]

3. PDF 输出需要区分“合并成单一连续 PDF”和“PDF 包 / Portfolio”。Adobe Acrobat 文档说明，合并文件可将 PDF、Word、Excel、PowerPoint、音频、视频或 Web 文件合并为一个 PDF，并可在合并前预览、排列、移动或删除页面；PDF 包则保留多个组成文件的独立身份，支持分类、排序、搜索、共享、打印和复用。对当前项目而言，一期宜输出单一连续 PDF，二期可考虑 PDF 包或 ZIP 包保留原件结构。[来源: https://helpx.adobe.com/cn/acrobat/desktop/edit-documents/combine-files/combine-files.html][来源: https://helpx.adobe.com/cn/acrobat/using/overview-pdf-portfolios.html]

4. 水印不只是页面视觉效果，而是材料外发控制的一部分。Adobe 将水印定义为嵌入文档的文本或图像，可用于所有权、保密性、品牌和防止未授权使用；当前项目应坚持服务端生成水印，并把申请人、用途、时间、包 ID、审批单号写入动态水印或隐性标识。[来源: https://www.adobe.com/acrobat/resources/what-is-a-watermark.html]

5. AI 推荐加入材料包应采用“AI 提取要求 / 推荐候选 + 人工确认 + 来源可追溯”的模式，而不是让 AI 直接生成最终包。ProposalDock 明确将 RFP 流程拆为需求与风险提取、已批准知识复用、人工 review/signoff 和导出；SEQUESTO 强调 AI 从已批准组织知识中草拟，答案可追溯到来源，每个审批被记录，最终判断仍由人负责。[来源: https://www.proposaldock.com/rfp-response-software][来源: https://sequesto.com/solutions/rfp-automation]

6. 对 API 消费和服务端生成，成熟模式是模板 + JSON 数据 + 输出格式控制。Adobe Document Generation API 使用 Word 模板和 JSON 数据生成 PDF 或 Word；TemplateFox 的 PDF API 使用 `template_id` 和 `data` 生成 PDF，可返回签名 URL、二进制 PDF、S3 存储位置，并支持 PDF/A；Carbone 开源引擎支持从模板和 JSON 数据生成 DOCX/PDF/XLSX 等格式。[来源: https://developer.adobe.com/document-services/docs/overview/document-generation-api][来源: https://templatefox.com/docs/api-reference/create-pdf][来源: https://carbone.io/documentation/developer/embedding/embedding-in-node.html]

## 五、推荐落地方案

### 5.1 一期可做

1. 建立 `certificate_package` 主对象：名称、场景、创建人、所属部门、共享范围、包类型、状态、模板 ID、材料数量、最后导出时间。
2. 建立 `certificate_package_item` 明细对象：证书 ID、文件版本 ID、排序号、材料分组、是否必选、加入方式（手动/筛选/标签/AI/预设包）、加入理由。
3. 支持四种加入方式：手动勾选、筛选结果加入、标签加入、AI 推荐后人工确认加入。
4. 支持三类预设包：华为项目投标包、政府采购标准包、公司能力宣传包；预设包可由管理员配置，不写死。
5. 导出前做三类校验：材料完整性、证书有效期/状态、用户权限。缺失项可以导出“草稿包”，但正式包必须通过校验。
6. PDF 导出一期采用单一连续 PDF：封面、目录、分组页、证书页。支持按模板排序，并允许导出前手动调整顺序。
7. 带水印导出默认可用；无水印导出触发审批。水印由服务端生成，并写入导出快照。
8. 每次正式导出生成版本快照：包内材料、材料版本、排序、模板、水印、审批状态、导出人、用途。
9. API 一期只开放内部消费：按包 ID 查询、创建包草稿、校验包、提交导出、查询导出任务、下载授权文件。

### 5.2 二期增强

1. AI 从招标文件、客户要求或聊天记录中提取资质要求，生成“要求-证书-证据”匹配矩阵。
2. 支持材料缺口任务分派：缺证书、缺授权、证书将过期、证书字段未核验时，自动分配给证书负责人或部门资质管理员。
3. 支持 DOCX 模板、PPTX 能力介绍模板、ZIP 原件包、PDF/A 归档包和 PDF Portfolio。
4. 支持包级协作：评论、任务、review、审批、锁定关键材料、导出前二次确认。
5. 支持历史包复用：按客户、行业、厂商、项目类型检索历史包，复制为新包后自动重新校验有效期。
6. 支持对投标助手开放稳定 API，并提供 webhook 或事件订阅：包生成完成、审批通过、材料过期、快照归档。

### 5.3 暂不建议

1. 暂不建议一期做完全自动生成正式投标包。行业来源都强调高风险文档需要人工 review/signoff，AI 只能推荐和草拟。[来源: https://www.proposaldock.com/rfp-response-software；https://sequesto.com/solutions/rfp-automation]
2. 暂不建议一期做外部客户可访问的材料包门户。当前权限、水印、审批和审计规则未定，先做好内部生成与审批。
3. 暂不建议把 PDF Portfolio 作为默认正式投标输出。Adobe 文档显示 PDF 包能保留原文件身份，但浏览器和普通 PDF 查看器兼容性存在限制，正式提交仍应以单一连续 PDF 为主。[来源: https://helpx.adobe.com/cn/acrobat/using/overview-pdf-portfolios.html]
4. 暂不建议把所有历史 Word/PDF 一次性批量转成正式证书包。应先完成资质对象、有效期、文件版本和权限校验，再做批量导入。

## 八、跨源矛盾检测结论

### 8.1 检测范围

- 已精读来源数量：外部网页/文档 18 个；脚本证据包成功精读投标文件摘录 3 个；另读取目标说明文档 1 个、本地既有 raw 调研 2 个。
- 检测对象：核心数字、日期、功能描述、因果判断、市场或公司事实。
- 检测时间：2026-06-09。

### 8.2 检测结果

结论：未发现影响本次推荐方案的硬性事实矛盾，但发现 4 类必须保留的差异和证据边界，不能静默合并。

### 差异项 1：单一 PDF 与 PDF 包不是同一输出形态

- 差异描述：Adobe 合并文件文档支持把多个文件合并为一个连续 PDF；Adobe PDF Portfolio 文档强调 PDF 包保留组成文件身份，便于添加、删除、分类、搜索和复用。
- 来源 A：https://helpx.adobe.com/cn/acrobat/desktop/edit-documents/combine-files/combine-files.html
- 来源 B：https://helpx.adobe.com/cn/acrobat/using/overview-pdf-portfolios.html
- 初步判断：二者是不同输出模式，不是矛盾。一期正式投标包建议用单一 PDF；二期原件归档或材料交换可用 PDF Portfolio/ZIP。

### 差异项 2：AI 自动化程度与人工审批边界

- 差异描述：SEQUESTO 强调 AI Agent 执行 intake、drafting、review、approval cycle，但也强调 bid managers retain full oversight；ProposalDock 明确 human review/signoff。
- 来源 A：https://sequesto.com/solutions/rfp-automation
- 来源 B：https://www.proposaldock.com/rfp-response-software
- 初步判断：AI 可用于提取、推荐、草拟和归因，但正式材料包生成必须有人确认，特别是无水印导出和正式投标提交。

### 差异项 3：厂商页面与可验证产品能力之间存在证据边界

- 差异描述：PandaDoc、ProposalDock、SEQUESTO、Flowcase 等页面能证明行业产品强调内容库、模板、审批、审计和导出，但公开页面通常不提供完整字段模型、权限矩阵、API 限流和部署方案。
- 来源：https://www.pandadoc.com/rfp-software/；https://www.proposaldock.com/rfp-response-software；https://sequesto.com/solutions/rfp-automation；https://www.flowcase.com/blog/12-features-to-look-out-for-when-evaluating-rfp-technology
- 初步判断：可作为需求方向和功能清单证据，不可直接作为实施承诺；架构阶段需 POC。

### 差异项 4：包级版本快照与文件级版本不是同一粒度

- 差异描述：SharePoint Document Sets 支持包级快照，SharePoint versioning 文档强调文件/列表项级版本历史；两者粒度不同。
- 来源 A：https://support.microsoft.com/en-us/office/create-and-manage-document-sets-c71d5796-d559-48de-b1b3-42383bdd13ea
- 来源 B：https://learn.microsoft.com/en-us/microsoft-365/community/versioning-basics-best-practices
- 初步判断：当前项目应同时保留文件版本和包快照。文件版本回答“这张证书何时变更”，包快照回答“某次正式导出用了哪些材料版本”。

2. 包内材料权限继承规则是什么：用户能看包但不能看包内某证书时，是否隐藏、脱敏、提示无权，还是禁止生成？

5. 包级审批触发条件是什么：无水印、全量资质、跨部门材料、外发客户、指定敏感证书是否都要审批？
6. 无水印文件下载有效期、下载次数、再次申请规则如何定义？

8. 预设包初始内容由谁维护？华为项目投标包、政府采购标准包、公司能力宣传包分别包含哪些必选/可选材料？
9. 有效期校验的阈值是什么：已过期必拦截，临期多少天提示，是否允许带风险说明导出？
10. API 消费方除投标助手外还有哪些？需要同步调用还是异步任务？文件返回二进制还是限时 URL？
11. 是否需要保留每次导出的不可变归档件？保留期限、存储位置和删除权限如何定义？
12. 证书包是否需要记录“用于哪个客户/项目/投标编号/合同编号”，以便后续追溯材料外发历史？

## 十、对 PRD / 架构文档的输入

### 10.1 PRD 输入

- 增加“证书包主对象”和“包内材料明细”需求，区分草稿包、待审批、已审批、已导出、已归档等状态。
- 增加“加入方式”需求：手动、筛选、标签、AI 推荐、预设包复制。
- 增加“导出前校验”需求：材料完整性、有效期、权限、水印/审批、文件格式。
- 增加“包模板配置”需求：分组、排序、必备材料、封面/目录、导出格式、审批规则。
- 增加“包版本快照”需求：正式导出、审批通过和 API 消费时自动生成快照。
- 增加“包级协作与共享”需求：共享范围、协作者、评论、任务、锁定内容。
- 增加“AI 推荐解释”需求：推荐理由、命中标签、引用证据、缺口与风险。

### 10.2 架构文档输入

- 文件生成服务建议独立成模块：合并 PDF、生成目录、加水印、模板填充、异步任务、失败重试、导出日志。
- 包快照应不可变：快照关联文件版本 ID，而不是仅关联当前证书 ID，避免后续证书更新影响历史导出。
- 服务端水印必须在文件层生成，不能仅前端叠加；建议水印内容可配置，并包含导出上下文。
- API 设计建议采用异步导出：`POST /packages/{id}/export` 返回 job，`GET /export-jobs/{id}` 查询状态，`GET /exports/{id}/download` 返回授权下载。
- 对投标助手开放的 API 必须做鉴权、权限过滤、审计、限流和字段脱敏。
- 文档模板可采用两条路线评估：商业 API（Adobe/TemplateFox 类）与内网自建（Carbone/LibreOffice 类）。涉密或内部资质文件优先评估内网自建。
- 存储需区分原始文件、水印派生文件、无水印审批文件、导出包文件、快照归档文件，并定义生命周期策略。
