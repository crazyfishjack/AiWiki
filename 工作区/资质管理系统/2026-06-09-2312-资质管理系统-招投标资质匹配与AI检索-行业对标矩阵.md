## 一、当前技术思路纪要中的想法

目标说明文档中的方案已经把资质系统定位为“全公司横向统一的资质数据底座 + 文件处理 + 权限审批 + AI 辅助检索 + API 服务”。与本次主题直接相关的部分主要集中在 `M2 上传识别与字段核验`、`M5 查询、AI 检索与证书包`、`M7 API 与系统集成`。

当前想法可概括为：

- 用户可上传 PDF / 图片证书，系统通过 OCR 提取资质名称、颁发机构、持有主体、发证日期、到期日期、证书编号、所属部门、证书负责人等标准字段，再由 AI 二次审核并提示低置信字段，最终由上传者人工核验后入库。
- 系统为资质记录生成语义标签，覆盖资质类型、业务场景、技术方向、资质级别，并支持人工补充资质背景说明、典型使用场景；联网检索只能生成候选内容，必须经人工确认后入库。
- 查询侧同时支持传统组合筛选和 AI 自然语言检索，允许用户粘贴招标文件资质要求、微信聊天记录或要求片段，系统先复述识别需求，再基于语义标签推荐资质清单，并支持一键加入证书包。
- 证书生命周期已考虑有效、即将到期、已过期、已归档等状态；普通资质依赖到期日期，软件著作权按资产型资质处理，不进入到期链路。
- 对外系统，尤其是投标助手，可通过 RESTful API 消费资质查询、自然语言匹配、详情和文件能力；联网检索是否启用、允许发送字段和权威来源标准仍是架构前待确认项。

行业对标后的判断：当前方向基本正确，但需要把“AI 推荐”进一步拆成可审计的工程链路：招标要求解析、硬条件门控、软条件评分、证据定位、缺口提示、人工确认、检索审计。否则容易变成黑盒推荐，难以支撑投标合规自查和后续争议复盘。

## 二、行业优秀做法归纳

### 2.1 招标要求粘贴 / 上传

国家发展改革委等部门在 2026 年招投标 AI 推广意见中明确提出，投标策划场景应“自动提取并解析项目招标计划、招标公告和招标文件等资料中的关键要素”，并生成招标需求图谱、提示重要内容。[来源: https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html]

行业工具和研究通常把输入分成三类：纯文本粘贴、PDF / 扫描件上传、结构化历史文档。面向长文档时，较好的做法不是直接把全文塞给大模型，而是先做 OCR / 版面解析 / 章节切分 / 表格提取 / 元数据保留，再进入检索和抽取链路。[来源: https://arxiv.org/html/2506.00608v2] [来源: https://www.mdpi.com/1996-1073/14/15/4632]

对当前项目的启发：一期至少支持“粘贴招标要求片段 + 上传招标文件 PDF”的两个入口。粘贴适合轻量试用；上传适合正式投标分析。上传文件必须保存解析版本、原文页码、段落位置和抽取证据，方便人工复核。

### 2.2 资格条件抽取

招投标文件解析的核心不是摘要，而是把“资格要求 / 否决项 / 评分项 / 证明材料 / 有效期要求 / 项目经验 / 人员要求”抽取成可计算对象。发改委意见提到招标文件检测应开展合规性、合理性、错敏词等多维度检测，自动识别违法违规和排斥限制竞争等问题，并提示判断依据和修改建议。[来源: https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html]

EPC 投标风险研究把 ITB 文件拆成数据库行，通过风险词库、phrase matcher、NER 和专家规则抽取风险条款，并输出原文句子、标签、风险组等结构化字段；其 PoC 中 CRC 模块和 TFA 模块分别达到约 92% 和 88% 的风险句抽取准确率，人工工程师约为 70% 和 86%。[来源: https://www.mdpi.com/1996-1073/14/15/4632]

对当前项目的启发：资质条件抽取结果应设计为结构化表，而不是一段 AI 文本。建议字段包括：条件原文、条件类型、硬/软属性、匹配对象、所需证明材料、证书名称/等级、有效期规则、数量阈值、是否可替代、来源页码/段落、抽取置信度、人工确认状态。

### 2.3 语义标签

采购与招投标研究中的常见做法是使用标准分类体系、历史文档和知识图谱来减少语义歧义。CPV 论文指出，公共采购分类存在层级深、细粒度类别样本稀少、标签不准确或过泛等问题，因此采用基于标签描述和层级结构的 zero-shot hierarchical classification 来改进标讯分类。[来源: https://arxiv.org/html/2405.09983v2]

工业采购评测 IndustryBench 进一步说明，工业采购知识必须受标准、产品记录、技术阈值和安全约束约束；构造数据时使用 GB/T 标准和结构化产品记录，并通过外部检索验证剔除大量看似合理但无法支撑的候选 QA。[来源: https://arxiv.org/html/2605.10267]

对当前项目的启发：语义标签不应完全自由生成。应建立“业务场景标签 + 技术方向标签 + 资质类别标签 + 证书等级标签 + 厂商/生态标签”的受控词表，同时保留同义词、别名和人工新增流程。

### 2.4 硬条件与软条件区分

公共机构采购报价验证论文强调，受监管场景中的自动判断必须可追溯、法律上可验证，不能只给二元分类结果；其方法把 LLM 抽取出的谓词值交给 Logic Tensor Network，用规则真值、谓词真值和文本证据解释最终判断。[来源: https://arxiv.org/html/2604.05539]

这对资质匹配尤其关键：硬条件是门槛，如“必须具备 ISO 27001 且证书在有效期内”；软条件是排序依据，如“类似项目经验较多”“技术方向更贴近”。硬条件未满足时不能通过软分补偿；软条件只能影响排序和推荐理由。

建议规则：

- 硬条件：资质名称/类型、等级、持有主体、证书状态、有效期、数量阈值、法律或招标文件明确的“必须/不得/否决”要求。
- 软条件：业务场景贴合度、技术方向相似度、历史使用频次、证书包适配度、客户行业相关性、补充材料完整度。
- 模型输出：先给“是否满足硬条件”，再给“软条件匹配分”，最后给“推荐理由”和“缺口提示”。

### 2.5 证书有效期约束

当前项目已设计有效、即将到期、已过期、已归档等生命周期状态。行业对标后建议把有效期作为硬门控：招标文件若要求“投标截止日 / 合同签订日 / 服务期内有效”，系统应明确计算基准日期并输出证据。若证书即将到期但仍满足投标截止日，推荐结果应标记风险，而不是直接剔除。

工业采购评测强调，工业答案若违反标准阈值或安全约束，即使部分正确也不可接受；其安全违规评分会把违反安全关键约束的答案归零。[来源: https://arxiv.org/html/2605.10267] 这个思路可迁移到证书有效期：过期证书、主体不符、硬性数量不足应视为不可推荐。

### 2.6 RAG / 向量检索 / 混合检索 / rerank

招标文件 RAG 论文提出在半结构化招标文档生成中使用历史招标文档库、字段级索引、embedding 检索、词汇索引和采购清单重排序；其医疗领域 1406 份历史招标文档实验中，框架总分 77.74，高于 ChatGLM-4 baseline 的 12.55 和仅带 retrieval module 的 29.42。[来源: https://arxiv.org/html/2410.09077v1]

工业需求工程 RAG 论文在真实汽车制造文档中使用混合语义 + 词法检索、cross-encoder rerank、源追溯和多模型编排；实验中 RAG 达到 98.2% 抽取准确率，Hybrid + Rerank 的 MRR 为 0.847，高于 BM25-only 的 0.683 和 Dense-only 的 0.791。[来源: https://arxiv.org/html/2603.20534v1]

PAKTON 合同分析框架使用结构化解析、层级 chunk、BM25 + dense embeddings + RRF、LightRAG、cross-encoder rerank，并把检索结果用于可解释报告；在 LegalBench-RAG 聚合结果中，PAKTON Recall@1 为 26.77%，高于最强基线 4.94%。[来源: https://arxiv.org/html/2506.00608v2]

对当前项目的启发：一期可不追求复杂 GraphRAG，但不建议只做向量检索。最低可采用“结构化硬过滤 + BM25 / 关键词 + 向量召回 + rerank + 权限过滤”的混合检索。

### 2.7 推荐理由、缺口提示与人工确认

发改委意见明确要求模型生成结论不替代招标人、招标代理机构、投标人、评标专家等主体的自主判断，不改变使用主体的法定责任。[来源: https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html]

PAKTON 的可借鉴点是输出结构化报告：主题摘要、推理过程、关键发现、证据来源、知识缺口和后续问题；同时明确标注无法支撑的结论，避免 unsupported claims。[来源: https://arxiv.org/html/2506.00608v2]

对当前项目的启发：推荐结果必须包含“为什么推荐 / 为什么不推荐 / 还缺什么 / 需要谁确认”。人工确认不是可选美化功能，而是 AI 结果进入证书包、投标助手或对外材料前的必要状态转换。

### 2.8 检索审计

工业 RAG 论文把 traceability 作为生产级设计原则，记录文档来源、版本、章节路径、页码、检索分数、生成模型、提示词、时间戳、token 使用和置信度，并获得专家最高评分 4.67/5.0。[来源: https://arxiv.org/html/2603.20534v1]

对当前项目的启发：检索审计应至少记录：用户输入、解析出的条件、参与匹配的资质 ID、检索 query、召回方式、BM25 / 向量 / rerank 分数、权限过滤结果、模型版本、推荐理由、缺口提示、用户确认人、确认时间、最终加入证书包动作。

### 2.9 外部联网检索边界

发改委意见强调安全可控、数据基础、模型安全、黑箱/幻觉/算法歧视风险防范，并坚持 AI 的辅助定位。[来源: https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html]

IndustryBench 的构造经验显示，LLM 生成候选内容即使经过前置筛选，也会在外部事实验证阶段被大量剔除；其 Stage 4 search-based verification 剔除了 70.3% 的前序幸存候选 QA，说明“看似合理”与“可验证”之间有明显差距。[来源: https://arxiv.org/html/2605.10267]

对当前项目的边界建议：外部联网检索只用于公开资料补充、同义词扩展、资质背景说明候选和厂商规则人工核验，不应上传客户未公开招标文件、内部证书文件、个人证书信息或投标策略。联网结果必须进入“候选态”，经人工确认后才允许入库或影响推荐。

## 四、推荐落地方案

### 4.1 一期可做

1. 招标要求输入：支持粘贴文本和上传 PDF；上传后执行 OCR / 文本抽取 / 章节切分，保留页码、段落和原文定位。
2. 条件抽取：把招标要求抽成结构化条件表，字段至少包括条件原文、类型、硬/软属性、证书类别、等级、数量、有效期、证明材料、来源位置、抽取置信度。
3. 资质库匹配：先用结构化字段硬过滤，再用关键词 / BM25 + 语义向量召回，最后 rerank。硬条件失败不能被语义相似度补偿。
4. 有效期与状态门控：过期、归档、主体不符、硬性数量不足直接进入“不满足”；即将到期但仍满足基准日期的资质进入“可用但有风险”。
5. 推荐结果解释：每条推荐给出匹配条件、匹配证书、来源原文、证书有效期、推荐理由、风险提示、缺口提示。
6. 人工确认闭环：推荐结果默认为“待确认”；用户确认后才能加入证书包或推送给投标助手。
7. 检索审计：记录输入、抽取条件、检索策略、召回结果、rerank 分、过滤原因、模型版本、人工确认动作。
8. 外部联网检索默认关闭或管理员手工触发；仅允许发送公开字段，如资质名称、证书名称、标准号，不发送客户招标全文、证书文件和个人信息。

### 4.2 二期增强

1. 构建资质知识图谱：连接资质、证书、厂商、人员、部门、项目经验、历史证书包、招标条件和成功案例。
2. 引入规则引擎或轻量神经符号层：把“必须具备、不得、有效期、数量阈值、主体一致”等硬条件做成可审计规则。
3. 支持招标文件版本比对：预公告、正式稿、澄清答疑之间做差异高亮，提示新增资格要求和否决项。
4. 支持自动生成合规自查报告：按招标要求列出满足/不满足/待确认，附证据页码和资质文件链接。
5. 建立领域评测集：用真实招标要求和人工标注的匹配结果评估抽取准确率、硬条件召回率、误推荐率、缺口提示准确率。
6. 联网检索白名单：对权威来源、厂商官网、标准库、政府平台做白名单检索，并把来源质量进入评分。

### 4.3 暂不建议

1. 暂不建议让 AI 自动决定是否投标或是否满足资格条件。政策和论文均强调高风险场景需要人工责任主体和可解释证据。[来源: https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html] [来源: https://arxiv.org/html/2604.05539]
2. 暂不建议把客户完整招标文件、内部证书文件或人员证书信息发送到外部联网检索接口。该边界涉及数据出网、商业秘密和个人信息风险。
3. 暂不建议一期做复杂 GraphRAG 或多 Agent 全流程，除非已有足够历史材料、知识图谱维护责任人和评测数据。PAKTON 类框架强调质量和解释，但也承认延迟、成本、结构解析泛化存在限制。[来源: https://arxiv.org/html/2506.00608v2]
4. 暂不建议只用向量相似度做资质推荐。工业 RAG 证据显示混合检索 + rerank 优于单纯 BM25 或 dense 检索，且资质硬条件需要结构化过滤。[来源: https://arxiv.org/html/2603.20534v1]

## 九、对 PRD / 架构文档的输入

### 9.1 PRD 输入

- 新增“招标要求解析”能力：支持粘贴/上传、解析任务、条件表、原文定位、用户纠错。
- 新增“资质匹配解释”能力：推荐清单必须展示匹配条件、匹配证书、推荐理由、缺口提示、证据来源和人工确认状态。
- 新增“硬条件/软条件”需求：硬条件做门控，软条件做排序；过期、归档、主体不符、数量不足不能被软分补偿。
- 新增“检索审计”需求：用户输入、解析、召回、过滤、rerank、模型输出、人工确认全链路留痕。
- 新增“联网检索边界”需求：外部检索默认候选态，白名单和字段出网规则需后台配置。
- 新增“评测与验收”需求：建立小型人工标注集，验收抽取准确率、硬条件漏判率、误推荐率、缺口提示准确率和人工确认完成率。

### 9.2 架构输入

- 文档处理层：PDF/OCR/版面解析/表格抽取/章节切分/页码定位。
- 结构化抽取层：LLM 或 IE 抽取条件表，配合规则校验和低置信字段提示。
- 资质匹配层：结构化硬过滤 + BM25 + 向量召回 + RRF / rerank + 权限过滤。
- 规则层：证书有效期、主体、等级、数量、状态、否决项等规则可配置、可审计。
- 证据层：保存招标原文位置、资质字段来源、检索分数、模型版本、prompt 摘要和人工确认动作。
- 安全层：外部联网检索网关、敏感字段脱敏、白名单来源、审计日志、失败降级策略。
- API 层：面向投标助手返回匹配结果、解释、缺口、证据和审计 ID，而不只返回资质列表。
