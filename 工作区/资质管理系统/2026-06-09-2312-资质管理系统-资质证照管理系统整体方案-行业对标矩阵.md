# 资质管理系统 · 资质 / 证照管理系统整体方案 · 行业对标矩阵

## 一、调研范围与结论摘要

### 1.1 本次调研问题

本次调研围绕“资质 / 证照管理系统整体方案”展开，重点看行业内同类系统、相邻问题和成熟实现方法，目标是为当前资质管理系统的 PRD 与架构设计提供外部证据支撑。

调研对象包括：

- 成熟 SaaS / 平台做法：企业证书、员工证书、供应商资质、职业资格和合规文档管理。
- 大厂实现方法：OCR 结构化识别、审批集成、文件水印 / 加密 / 权限保护。
- 社区或开源最佳实践：W3C Verifiable Credentials、OpenCerts / TrustVC、PaddleOCR 文档智能。
- 行业标准：国内电子证照国家标准体系、W3C VC 数据模型。

### 1.2 当前技术思路纪要中的想法

当前技术思路纪要将系统定位为“全公司横向统一资质管理平台 + 资质数据底座 + 对外 API 服务”，而不是单一文件管理工具；核心模块包括资质资产管理、上传识别与核验、生命周期与到期提醒、权限 / 水印 / 下载审批、查询与 AI 检索、证书包、厂商依赖告警、统计仪表盘、Logto / 钉钉 / 投标助手 / Atlas 集成、初始数据导入与全链路验证。该判断来自本项目技术思路纪要的模块草案、能力映射和数据流转设计。[来源: `工作区/资质管理系统/2026-06-09-1834-资质管理系统-技术思路纪要.md`]

当前方案已经覆盖行业主线：集中存储、有效期提醒、权限隔离、审批下载、OCR 入库、证书包组合、API 消费和审计风险控制。与成熟实践相比，还需要加强四点：第一，资质记录要显式建成“证据对象”，包含来源、字段置信度、人工确认、版本与历史；第二，水印和无水印下载不能只靠前端控制，应使用服务端生成、权限控制和审批留痕；第三，AI/OCR 识别应以“候选字段 + 置信度 + 人工核验”进入主流程；第四，对外 API 应先提供稳定的查询 / 匹配 / 详情 / 文件授权接口，再考虑复杂语义检索。

已按 Wiki 调研技能读取 `SCHEMA.md` 和 `wiki/_index.md`，并检索 `资质 / 证照 / 证书 / OCR / 审批 / 水印 / 生命周期 / 到期提醒` 等关键词。Wiki 中未发现资质 / 证照管理系统的直接知识页，仅存在业务理解、需求挖掘、专家判断提取等方法论页面，可作为需求分析方法参考，但不足以回答行业对标问题。因此继续执行标准四工具联网调研流程。

## 二、行业优秀做法分层分析

### 2.1 成熟 SaaS / 平台做法

成熟 SaaS 对“企业证书、许可证、职业资格、供应商资质”的共性实现通常不是从文件夹开始，而是从“可运营的合规记录”开始：集中记录库、有效期字段、责任人、状态看板、自动提醒、自助更新、审计报告、外部共享和导出。Aptien 明确将公司证书、执照、许可、职业考试放在中央仓库，并提供过期 / 续期提醒、受控访问、任务协作、审批、文档归档、报表和自定义字段。[来源: https://aptien.com/en/certification-management-software]

Expiration Reminder 的认证跟踪方案强调以实时仪表盘管理凭证数据库、自动通过邮件 / SMS / WhatsApp 发送提醒、生成可定制报告、设置合规规则、支持表格导入和系统集成，并声明 HIPAA、SOC 2、GDPR 合规。[来源: https://www.expirationreminder.com/solutions/certification-tracking-software]

OneCredential 的供应商资质管理更贴近“对外协作”：它把营业执照、保险证书、DEI / diversity credentials、所有权声明、UBO IDs、制裁与风险检查等放入 centralized credential vault，并提供供应商自助上传、到期提醒、实时状态看板、QR code / shareable link、审计导出和重新验证流程。[来源: https://www.onecredential.io/products/credential-management]

Certemy / Carahsoft 的许可证和认证管理材料显示，成熟平台会把 license lifecycle、license tracking、primary source verification 拆成相互衔接的能力：配置化流程、自动提醒、文档上传、HRIS / SSO 集成、实时合规仪表盘、按员工角色 / 证书类型 / 地点 / 区域跟踪，以及从 governing boards 自动验证证书状态、生成时间戳截图和审计轨迹。[来源: https://www.carahsoft.com/certemy/solutions/complete-license-and-certifications-management]

对当前项目的借鉴：一期不应只做“证书文件 CRUD”，而应把“谁负责、何时到期、如何确认、何时提醒、是否可见、能否下载、是否被下游消费、历史版本在哪”作为同一个生命周期闭环。

### 2.2 大厂实现方法

阿里云 OCR 的营业执照识别 API 能提取统一社会信用代码、公司名称、地址、主体类型、法定代表人、注册资金、成立日期、营业期限、经营范围等结构化字段，并返回字段置信度、坐标、图案信息；它也明确提示图片质量、尺寸、格式、大小会影响识别效果，且该接口暂不支持 PDF。[来源: https://help.aliyun.com/zh/ocr/developer-reference/api-ocr-api-2021-07-07-recognizebusinesslicense]

这对当前项目有两个直接影响：第一，证照 OCR 入库应把字段候选值、字段置信度、坐标、文件质量和人工修正记录保留下来；第二，PDF 证书如果要走 OCR，架构上需要先做 PDF 转图片 / 页图提取，不能假设所有 OCR API 原生支持 PDF。[来源: https://help.aliyun.com/zh/ocr/developer-reference/api-ocr-api-2021-07-07-recognizebusinesslicense]

钉钉开放平台的审批托管机制支持三方业务在创建审批模板 / 实例时注册自定义按钮渲染和审批操作回调，`CUSTOM_ACTION_DEFINITION` 用于回调三方获取按钮定义，`CUSTOM_ACTION_APPLY` 用于回调三方处理审批提交操作；审批详情页可复用分享、下载、打印、催办、评论、建群等通用按钮，也可由三方系统定义同意、拒绝、转交、加签、退回等行为。[来源: https://developers.dingtalk.com/document/development/detailed-explanation-of-hosting-interface-agreement-in-approval-operation-area]

这支持当前项目把“无水印下载审批”设计为系统内申请单 + 钉钉审批任务 + 回调闭环，而不是简单发一条消息；系统必须保存 `processInstanceId`、`taskId`、审批动作、审批人、回调结果、限时下载链接和失败补偿状态。[来源: https://developers.dingtalk.com/document/development/detailed-explanation-of-hosting-interface-agreement-in-approval-operation-area]

Microsoft Purview sensitivity labels 显示，大厂文件保护通常同时做分类、内容标记、水印、加密和权限控制；敏感标签可以给文档加水印、页眉、页脚，也可以通过加密限制特定用户 / 组的访问和可执行动作。[来源: https://learn.microsoft.com/en-us/purview/sensitivity-labels]

Microsoft 的加密标签还支持内容访问到期、离线访问限制、打开时重新认证、权限组控制等策略；动态水印会在用户访问文件时插入用户 UPN / 邮箱等标识，官方说明该水印比普通静态标记更难被用户手动移除，但导出格式和权限仍会影响保护效果。[来源: https://learn.microsoft.com/en-us/purview/encryption-sensitivity-labels]

这对当前项目的借鉴是：带水印文件应优先服务端生成或通过文档保护服务生成；无水印文件应作为审批后的派生文件，绑定有效期、下载次数、申请单、审批记录和访问日志。

### 2.3 社区 / 开源最佳实践

PaddleOCR 的 KIE 文档把“文字识别”与“关键信息抽取”明确拆开：先由 OCR 模型提取文本和位置，再由 KIE 模型根据图像、文本位置和文本内容抽取关键字段；KIE 可使用 SER 识别 key/value 类别，也可用 SER + RE 关联问题和值。[来源: http://www.paddleocr.ai/v2.9.1/en/ppstructure/blog/how_to_do_kie.html]

PaddleOCR 文档还指出固定版式场景可少量标注起步，文本检测建议对特定场景做 fine-tuning，文本识别在证件、发票等场景可能受罕见字和特殊字体影响，必要时需基于特定文档场景数据集微调。[来源: http://www.paddleocr.ai/v2.9.1/en/ppstructure/blog/how_to_do_kie.html]

OpenCerts v3 属于新加坡 Smart Nation 相关的电子证书开源实践，它基于 TrustVC，支持 W3C Verifiable Credentials 和 OpenAttestation Verifiable Documents，提供签发、验证、选择性披露能力，并使用 `did:web` 标识签发方。[来源: https://docs.opencerts.io/docs/]

OpenCerts 的可借鉴点不是让当前项目一期上链，而是给出“证书数据可信表达”的思路：签发方、有效期、状态、主体、证明、验证流程应可结构化表达；对外共享证书包时，可以先设计可验证元数据和审计记录，后续再评估是否需要 VC / 签名 / 防篡改能力。[来源: https://docs.opencerts.io/docs/]

### 2.4 适用行业标准

W3C Verifiable Credentials Data Model v2.0 将可验证凭证定义为由 issuer 对 subject 作出的一组 claims，并描述 issuer、holder、verifier 三方生态；核心属性包括 `issuer`、`credentialSubject`、`validFrom`、`validUntil`、`status`、`credentialSchema`、`evidence` 等。[来源: https://www.w3.org/TR/vc-data-model-2.0/]

W3C VC 还强调“可验证”并不等于声明内容天然为真；验证方在确认凭证真实性和当前有效性后，还要根据自身业务规则评估 issuer、proof、subject 和 claims。[来源: https://www.w3.org/TR/vc-data-model-2.0/]

国内电子证照国家标准体系方面，公开资料显示《电子证照 总体技术架构》等六项国家标准于 2018 年发布，并于 2019 年 1 月 1 日实施；相关标准覆盖总体技术架构、目录信息、元数据、标识、文件技术要求和共享服务接口规范。[来源: http://www.cac.gov.cn/2018-11/17/c_1123726179.htm]

国家数字标准馆页面显示，GB/T 36901-2018《电子证照 总体技术架构》规定电子证照系统技术框架、电子证照资源库、共享服务要求和安全要求，适用于电子证照系统的规划、设计、开发和利用。[来源: https://www.ndls.org.cn/standard/detail/7401ca73dfed015abe2b98a8d52ca830]

对当前项目的现实建议：一期做内部资质管理，不必完全实现政务电子证照平台；但字段模型、标识、元数据、共享服务接口、安全要求应参考该类标准，尤其是“资源库 + 共享服务 + 安全要求”的分层。

## 四、推荐落地方案

### 4.1 一期可做

| 方向 | 建议落地内容 | 依据 |

## 七、对当前方案的结构化建议

### 7.1 产品能力建议

| 模块 | 建议 |
| --- | --- |
| 资质资产管理 | 把资质作为主数据对象，而不是附件对象；每条资质应有关联文件、字段、来源、状态、版本、责任人、部门、使用场景、标签和审计记录。 |
| 上传识别 | 文件上传后进入“识别中 / 待核验”，OCR 结果不能直接变为有效数据；每个关键字段保留候选值、置信度、人工确认人、确认时间。 |
| 生命周期 | 到期状态、续签、归档、离职人员证书归档和厂商依赖风险要统一进入状态机，不要散落在提醒任务里。 |
| 权限与安全 | 权限判断应同时覆盖列表、搜索、详情、下载、导出、证书包、API；普通员工搜索跨部门资质时不能暴露存在性。 |
| 水印与审批 | 带水印文件作为默认派生文件；无水印文件作为审批后派生文件，限时下载并记录审计。 |
| 检索与证书包 | 传统筛选作为基础能力，AI 检索作为增强；AI 推荐必须可解释、可修正、可加入证书包。 |
| API | 先做稳定 REST 能力和文件授权，语义匹配作为独立接口；API 调用必须有消费方、用途、字段范围、权限、限流和日志。 |
| 统计看板 | 看板不只统计数量，还要突出风险：即将到期、已过期、待核验、无负责人、低置信字段、厂商依赖不足、审批超时。 |

### 7.2 架构建议

建议采用分层：

```text
Web / Atlas 入口
  -> 应用服务层：资质管理、上传核验、生命周期、审批下载、证书包、统计、API
  -> 领域层：资质、文件、字段、状态、提醒、权限、审批、证书包、厂商依赖、审计
  -> 集成层：Logto、钉钉网关、OCR/AI、文件存储、水印服务、投标助手 API、联网检索
  -> 数据层：结构化数据库、对象存储、搜索索引、审计日志、任务队列
```

关键架构约束：

- 文件服务和业务库要分离；数据库存元数据、对象存储存原始文件和派生文件。
- OCR/AI 结果要可重跑、可追溯，不要覆盖人工确认后的字段。
- 状态计算和提醒应通过任务队列 / 定时任务执行，并记录每次提醒事件。
- 无水印下载链接必须短期有效，过期后重新申请或重新授权。
- 水印生成应在服务端完成，前端只负责预览和下载入口。
- API 文件获取必须二次鉴权，不能因为调用方系统有权限就跳过资质系统自身权限规则。

## 九、对 PRD / 架构文档的输入

### 9.1 对 PRD 的输入

- PRD 应把“资质记录”定义为核心对象，文件只是其证据附件之一。
- PRD 应增加字段级 OCR 置信度、人工确认、字段来源、历史版本和核验状态。
- PRD 应输出权限矩阵，覆盖列表、搜索、详情、预览、带水印下载、无水印申请、证书包共享、API 调用。
- PRD 应把无水印下载设计成流程：申请、审批、回调、生成、限时下载、过期、审计。
- PRD 应定义到期提醒规则：提醒对象、提醒天数、升级链、已处理条件、失败重试、通知记录。
- PRD 应把证书包定义为可保存、复用、共享、导出、审批的业务对象。
- PRD 应定义初始导入验收：真实样本覆盖 OCR、人工核验、状态流转、提醒、权限、水印、审批、导出和 API。

### 9.2 对架构文档的输入

- 架构文档应明确文件存储、派生文件、水印生成、下载鉴权和链接过期策略。
- 架构文档应明确 OCR/AI 服务边界：云 OCR / 私有 OCR、PDF 前处理、字段置信度、人工核验、重跑机制。
- 架构文档应明确 Logto 与钉钉 ID 映射，尤其是审批人、提醒对象和组织链。
- 架构文档应明确钉钉审批回调的状态机、幂等、失败补偿和审计字段。
- 架构文档应明确对外 API 的鉴权、授权、字段脱敏、限流、调用日志和版本策略。
- 架构文档应保留 W3C VC / OpenCerts 的远期扩展点，但一期不引入完整 DID / VC / 上链实现。
- 架构文档应设计搜索索引：结构化字段检索优先，AI 语义匹配作为增强，不替代权限过滤。
