## 0. 导读

第二章描述的是**一条完整产品主线**，用户沿着这条主线即可从“拿到一堆原始文件”走到“导出一份合规、可追溯、可交付的正式文档”。整条主线分为九个环节：

```text
① 创建文档类型包（定义“这类文档怎么做”）
  → ② 文件解析与要素抽取（读懂原始资料 + 抽外部需求清单）
  → ③ 技能库（把专家经验/规则沉淀为可挂载的提示词与审查规则）
  → ④ 排版模板（用可视化配置块定义最终样式）
  → ⑤ 锁定解析结果（生成历史版本，进入下一阶段）
  → ⑥ 章节目录规划（机器可读结构 + 资料需求 + 责任人 + 审查清单）
  → ⑦ 分章节内容生成（识别依赖、缺料不编造、可编辑可重生成）
  → ⑧ 构建器导出（样式独立于生成，导出即留版本）
  → ⑨ 资产库 + 全程协同（可并发、防冲突、可追溯）
```

贯穿全程的三条**不变量**（来自调研的共识结论，后文反复出现）：

1. **证据可溯源**：每一个抽取出的字段、每一句事实表述、每一条风险，都必须能点击跳回原文的具体行/段/表格单元格。结构正确不等于事实正确，没有原文锚点的结论不允许进入“已确认”状态。
2. **人机边界清晰**：AI 负责“召回、解析、解释、起草、查漏”；最终“是否满足、是否偏离、报价、法律责任、最终提交”由人确认。高风险对象默认进入人工复核队列，不得自动通过。
3. **缺料不编造**：生成阶段一旦资料不足，输出“缺资料清单”并以风险形式显示，绝不凭空编造资质、人员、业绩、参数或承诺。

## 1. 文档类型包（DocumentTypePackage）

### 1.1 设计目标

“文档类型包”是整个平台的**根抽象**，也是平台能从“投标助手”长成“通用文档工厂”的关键。它回答一个问题：**这一类文档应该抽什么、怎么判断、必须有什么证据、抽完如何进入后续规划与审查**。

未来要新增标书、申报书、白皮书、可研报告、合同审查报告等文档类型时，应当**只新增/调整一个文档类型包**，而不是重写一套解析与生成逻辑。

### 1.2 用户自定义的内容

用户在创建一个文档类型包时，至少需要配置：

- **名称**：如“投标文件”“科技项目申报书”。
- **说明（适用范围）**：适用行业、适用场景、适用风险等级，用于后续路由和提示用户。
- **抽取模板（核心）**：声明“要抽什么”，即一组**标签（字段）定义**。模板可基于内置模板派生、修改、另存为新模板，支持版本化管理。当配置抽取模板时，需要导入输入材料
- **可挂载的技能**：从技能库（见第 3 节）挂载若干技能，作为后续解析/规划/生成/审查时注入 LLM 的上下文。

## 2. 文件解析与要素抽取

这是整条主线的入口。调研的共识是：**成熟标书助手的核心不是“写”，而是“读标解析”**——先把外部要求、约束、评分/审查标准、评分点、废标项、交付格式抽成结构化对象，后续才能规划、生成、审查。

### 2.1 支持的输入格式与“原生解析优先”原则

平台需支持解析：Word、PDF、扫描件、图片、Excel、PPT、txt、md。

**关键设计：不要无差别 OCR。** 对“不用 OCR 就能更可靠解析”的文档类型，一律优先走原生解析：

| 输入类型 | 优先策略 | 原因 |
| --- | --- | --- |
| Word / DOCX | 原生解析 | 保留标题、样式、表格、批注、页眉页脚 |
| Excel / 清单 | 原生解析 | 保留 sheet、公式、单元格、合并单元格 |
| 可复制 PDF | PDF 文本层 + 版面分析 | 避免 OCR 错字，保留原文 |
| 扫描 PDF / 图片 | OCR + 版面分析 | 无文本层，只能识别 |
| 图纸 / 图片化表格 | 多模态 OCR + 人工复核 | 高风险，需保留图像证据 |
| 网页公告 | HTML/DOM 解析 | 表格、链接、附件更准确 |
| 电子平台导出 JSON/API | 直接入库 | 不要转文本再抽 |

### 2.2 解析结果：多表示保真，而非只留 Markdown

文档解析完成后再交给 LLM。这里有一个**容易踩的坑**：只把全文压成 Markdown 当唯一真相源。Markdown 适合给 LLM 阅读、适合人工预览、适合简单表格；但它会压扁复杂表格（合并单元格、跨页表头、行列层级），也无法表达页码坐标和图片区域，一旦作为唯一中间表示，后续就无法精确溯源。

因此解析层应**同时保留多种表示**：

- `原始文件`：原文件本身。
- `页面图片`：每页图片/截图，作为图像证据。
- `版面 JSON`：块、bbox 坐标、页码、阅读顺序、标题层级。
- `Markdown 视图`：给 LLM 和人阅读。
- `表格 HTML / 表格 JSON`：复杂表格用 HTML/结构化 JSON 保真。
- `语义分块`：按章节/语义切分的 chunk。
- `证据索引`：原文位置索引，支撑点击回跳。

### 2.3 LLM 自动分析：按模板抽取 + 自动识别新标签

文档解析后交给 LLM，按文档类型包里的**抽取模板**进行解析。LLM 的职责是：

1. **按模板抽取**：把内容划分到预定义标签中。
2. **自动识别模板外的新标签**：依据所挂载技能包中的描述，发现模板未覆盖但有价值的内容，自动建议为新标签，并把内容归入。
3. **按照技能中的描述**：根据技能给定的规则和要求进行抽取
4. **最终产出两类结果**：
   - **标签 + 内容**（既包含表格，也包含文字事实表述）。
   - 一份**外部需求清单**（说明编写文档时还需要哪些外部文件/材料）。
1. 根据技能确认解析的这些文件包是否完整，是否缺少其他材料

### 2.4 解析结果界面：左右联动 + 高亮可溯源 + 可编辑

这是第二章里用户体验最重的一块，要求如下：

- **左右联动布局**：页面左侧显示解析出的**完整原文文档**，右侧显示**最终生成的结构化内容**。
- **每个值/每条事实可点击溯源**：解析出的表格里的每个值、文字表述中的每个事实表述，都要**高亮可点击**（可对应多段原文）。点击后：
  - 弹出**原文表述列表**（一条结论可能对应多处原文，用于定位）。
  - 自动跳转到原文对应的**行/段落/表格单元格**并高亮。
- **生成结果可修改**：用户可直接修订右侧结构化结果，修订要保留“原始抽取值 + 人工修正值”，便于审计与回归。
- **外部需求清单同理**：以**列表**形式展示“编写文档时可能要用到哪些外部文件/材料”；点击每个条目，同样能跳回原文，告诉用户“招标文件在哪里说需要这份材料”。

## 3. 技能库（Skill Library）

技能库是把**专家经验、行业规则、企业偏好**沉淀为可复用资产的地方。一个技能本质是“给 LLM 的一段结构化上下文 + 一组审查规则”，可挂载到文档类型包内，在解析/规划/生成/审查阶段被注入。

### 3.1 技能编辑页面：六个标准输入框

技能页面提供六个输入框，用户选填后由程序组合，作为上下文注入 LLM：

| 输入框  | 作用                 |
| ---- | ------------------ |
| 背景   | 项目/行业/场景背景信息       |
| 任务   | 本技能要 LLM 完成什么      |
| 要求   | 内容、风格、格式上的硬性要求     |
| 模板   | 章节/表格/段落等可复用结构     |
| 特殊规则 | 明标暗标、行业特例、企业偏好等    |
| 审查规则 | 生成审查清单的规则依据（见 3.3） |
| 响应矩阵 | 响应矩阵要求             |


---

## 4. 排版模板（Word / Markdown）

排版模板用**可视化配置**而非让用户写样式代码：

- **可拖动的配置块**：文字、图片、表格、流程图四类块，可在画布上拖动、缩放。
- **三页示例**：用户用三页配置出整体风格。
- **程序可读对象**：这三页必须是结构化、机器可读的对象，平台据此**识别用户想要的整体风格**，后续排版与生成沿用这套模板。
- 支持 Word 与 Markdown 两种目标格式。

> **说明**：排版模板的“样式定义”与第 8 节“构建器”是同一思想的两端——**样式与内容解耦**。用户在此定义“长什么样”，构建器负责“把已确认内容套进这个样子并导出”。

## 5. 锁定解析结果，进入规划阶段

解析完成并经人工确认后，用户点击“**锁定当前解析结果**”：

- 锁定即**生成解析历史版本**，可回溯。
- 锁定后进入“生成大纲（章节目录）”环节。

## 6. 章节目录规划

目录不是普通目录，而是“**评审导航 + 作战图**”：它决定责任分工、素材清单、生成顺序和后续页码索引。

### 6.1 结构计划必须是机器可读对象

章节目录本身是结构化对象，至少包含：

- **多级目录**（几级标题、层级关系）。
- 每个目录有一句话，说明这个章节写什么，响应的是原文哪一条
- 每个目录下的**资料需求清单**。
- 每个目录可**指派责任人**。
- 每个目录可**人工指定使用的上下文**：
  - **当前解析文件**：用解析结果的哪一部分作为上下文。
  - **外部文档**：根据挂载的上下文包判定本节用哪个文档（直接粘贴的材料或参考文档）。
  - 若挂载的上下文包无合适资料，标记为“**外部获取**”，并给出建议，需要什么资料。
  - 生成时先由 LLM 自行判断上下文，再允许人工调整。

### 6.2 目录操作

- 上移、下移、修改、删除。
- 前方新增同级、新增子目录。

### 6.3 目录阶段同时产出两张关键表

1. **响应矩阵**：抽取出来的“要求”与目录双向追溯，并标注每个目录可能存在什么风险。
2. **审查清单**：依据所挂载技能中的审查规则自动生成，**人可编辑**。


### 6.4 锁定章节，进入生成阶段

章节目录确认后，点击“**锁定当前章节**”，生成章节历史版本（可回溯），进入内容生成环节。

## 7. 分章节内容生成

核心原则：**长文档分章节生成、逐章审查确认，绝不一次性整本生成**；并且**缺料不编造**。

### 7.1 依赖识别与可视化

- llm识别**目录依赖**：哪些章节可能依赖之前章节生成的内容（避免前后矛盾）。
- 在章节前显示一个**小灰点**，鼠标悬停显示“依赖了哪些章节”。
- **可直接执行**（依赖已满足）的章节标绿，可点击“生成内容”。

### 7.2 单章生成与编辑

- **每个目录单独生成内容**。
- 界面布局：左侧目录树（可点击跳转），右侧当前章节内容。
- 上方仍需显示这个章节写什么，响应的是原文哪一条，需要哪些资料，责任人，使用的上下文（已经在上一个阶段锁定，但仍然可修改）
- **生成时识别缺失**：输出“缺资料清单”，**不得编造**；生成后以**风险形式**显示在内容下方。
- 生成内容**可被人编辑**；底部可输入**备注**（仅给人看，不进入正文）。
- 内容上方提供“**重新生成建议**”输入框：用户输入的建议作为**新增上下文**，点击后方“重新生成”按钮即可重新生成该章节内容。需要能获取到当前内容框

### 7.3 审查
根据审查清单和已生成的内容，根据技能中的说明，对文档已有内容进行审查

## 8. 构建器（Builder）与导出

**构建器必须独立于生成器**：生成负责“写什么”，构建负责“长什么样、怎么导出”。

构建器负责：

- **导出样式**：
  - 文档格式（Word 或 Markdown）。
  - 标题样式、页面格式（上下左右边距）、对齐方式、编号样式、正文样式、表格样式等。
  - 不同级别标题的字体、字号、编号样式。
- **导出即留版本**：导出后形成历史版本，保存当前所有章节内容，可回溯。

> **说明**：构建器与第 4 节排版模板配合——排版模板提供“风格定义”，构建器在导出时把“已确认的章节内容”套进该风格。两者解耦后，同一份内容可导出为不同样式/格式而无需重新生成。

## 9. 资产库（Knowledge / Asset Library）

资产库把“上传文件”升级为“**可治理资产**”，是生成真实内容（而非通用空话）的前提。

当前至少包含：

- **资质文件（图片）**：图片类资产要有详细文字说明，便于检索与直接引用。
- **需求上下文（原始文件，手动分类）**。

## 10. 全程协同与并发架构

所有环节都要求**可共同编辑**：

- 创建者在开始文档解析**之前**，需指定**当前项目负责人**（包括自己），并可配置其他负责人。
- 被配置到的用户可以查看项目、参与共同编辑。
- 因此，以上所有操作必须设计为**可并发、防冲突**的架构。
